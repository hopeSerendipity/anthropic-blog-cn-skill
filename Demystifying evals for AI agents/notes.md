# Demystifying evals for AI agents - 笔记

发布时间：`2026-01-09`

## 文章主旨

这篇文章主要讲的是，AI agent 的评估不能再沿用早期单轮 LLM 任务那种简单、静态的测试思路，因为 agent 会跨多轮调用工具、修改环境状态、根据中间结果调整策略，错误会传播，成功路径也可能非常多样。Anthropic 的核心观点是：`eval` 应该被当成 agent 工程体系里的基础设施，而不是发布前的附属检查。一个好的评估体系，需要围绕任务、trial、grader、transcript、outcome、harness 和 suite 这些基本对象来设计，同时结合确定性评分、LLM 评分和人工校准，才能真正帮助团队加速迭代、发现回退、验证升级，并把“感觉 agent 变差了”转化成可测量、可修复的问题。

## 核心观点

### 1. Agent 越有用，评估就越难，但也越必要

文章一开始就指出，agent 的价值恰恰来自它的自主性、灵活性和多轮执行能力，而这些特性也让评估复杂度大幅上升。普通单轮问答只需要看输出对不对，而 agent 需要看它在环境中的最终结果、过程中的工具使用、任务完成路径，以及多轮行为是否稳定。因此，越是强的 agent，越不能只靠人工试用和直觉判断，必须尽快建立系统化 eval。

### 2. 评估的对象不只是“回答”，而是完整任务执行系统

文章给出的一组定义非常关键：task、trial、grader、transcript、outcome、evaluation harness、agent harness、evaluation suite。它想说明的是，评估一个 agent，不是在评估某次回复文本，而是在评估“模型 + scaffold + 工具 + 环境”这个完整系统能否稳定完成任务。尤其重要的是区分 `transcript` 和 `outcome`，因为 agent 口头上说任务完成了，并不代表环境里的最终状态真的发生了变化。

### 3. Evals 的价值会复利增长，是 agent 团队的加速器

没有 eval 的团队往往陷入被动修 bug 的循环，只能等用户抱怨，再人工复现、修补，然后担心是否引入了新回退。有了 eval 之后，失败案例会被沉淀成测试任务，回归可以自动被发现，模型升级也能更快做出决策。文章强调，eval 不只是质量护栏，还是产品团队和研究团队共享目标、统一优化方向的沟通介质。

### 4. 不同 agent 类型可以共用一套评估思维，但 grader 组合要因任务而变

文章将 agent 粗分为 coding、conversational、research 和 computer use 四类，并说明它们虽然应用场景不同，但底层评估方法可以复用。coding agent 更适合用确定性测试验证最终正确性；conversational agent 更依赖状态检查和互动质量 rubric；research agent 需要 groundedness、coverage 和 source quality 的组合；computer use agent 则要在真实 GUI 环境中检查最终状态。共通原则是：为任务选合适的 grader，而不是为所有任务套同一种评估模板。

### 5. 优先评估结果，而不是过度约束过程

文章明确反对过于死板地检查 agent 是否严格按照预定顺序调用工具。原因是强模型经常会找到评估设计者没预料到、但同样有效的路径。如果 eval 只接受一种过程，就会把创造性和有效变体错误地判成失败。更好的做法通常是优先检查 agent 产出的结果是否满足要求，再在确有必要时补充一些对过程的约束。

### 6. 非确定性是 agent eval 的核心现实，必须用多 trial 和合适指标理解它

Agent 在不同运行间会发生波动，所以单次结果很难说明真实能力。文章用 `pass@k` 和 `pass^k` 把这个问题讲得很清楚：前者衡量“多试几次至少成功一次”的概率，适合那些只要出现一次可行解就有价值的系统；后者衡量“连续多次都成功”的概率，更贴近用户对稳定性的期待。这个区分很重要，因为很多团队会误把偶然成功当作系统可靠。

### 7. 从真实失败开始构建 eval，比等待“大而全”套件更有效

Anthropic 给出的路线图里，一个非常务实的建议是尽早开始，而且一开始不需要几百个任务，20 到 50 个来自真实失败案例的 task 就足够起步。先把开发中本来就要手动检查的内容、生产中的 bug 报告和支持案例转成测试任务，再逐步扩展。这个建议背后的逻辑是：早期改动效应量大，小样本也能提供强信号，而越晚开始做 eval，越容易沦为对已上线系统的反向解释。

### 8. 高质量 eval 的关键是任务清晰、环境稳定、grader 可校准

文章反复强调，很多低分其实不是模型不行，而是 task 写得含糊、环境有共享状态、grader 配置有 bug。一个好的任务应当让两个领域专家独立判断时也能给出相同结论；一个好的 eval harness 应该让每次 trial 从干净环境开始；一个好的 LLM grader 必须和人类判断持续校准，并允许在信息不足时返回 `Unknown`。换句话说，eval 本身必须像产品一样被设计、调试和维护。

### 9. 读 transcript 是做 agent eval 不可替代的核心动作

文章特别强调，不能只看分数，不读 transcript。只有读 transcript，团队才能知道某次失败到底是 agent 真犯错了，还是 grader 错杀了一个合法解；也只有读 transcript，才能发现隐藏在评估系统内部的规格歧义、路径偏见和评分漏洞。这一点几乎是全文最实用的建议之一，因为很多评估失真问题都不是靠聚合指标能看出来的。

### 10. Capability eval 和 regression eval 需要同时存在，并动态转换

能力型 eval 用来衡量“现在还做不到、但希望未来做到”的事情，所以理想状态是通过率不高、能提供清晰爬坡目标；回归型 eval 则用来守住已经做到的能力，所以应该接近 100% 通过。随着 agent 提升，一些能力型 eval 会逐渐“毕业”为回归套件。这个框架很有价值，因为它把“追求进步”和“防止退步”明确拆成了两类不同的评估职责。

### 11. Eval 不是孤立手段，而是整体质量体系的一层

文章最后把自动化 eval 放进一个更完整的质量框架里，与生产监控、A/B 测试、用户反馈、人工 transcript 审查和系统化人工研究并列。它借用了瑞士奶酪模型来说明，没有任何一种方法能单独挡住所有问题。真正成熟的团队，是把这些手段叠在一起使用：自动化 eval 提供快速迭代，生产监控给真实反馈，人工评审负责校准和补盲。

### 12. 最好的 eval 往往由最接近用户和需求的人共同塑造

文章最后一个很重要的观点是，eval 不应只属于研究团队或基础设施团队。最接近用户和产品需求的人，例如产品经理、客户成功经理、销售人员，其实最知道什么算成功、哪里最容易出错，因此也最适合贡献 eval task。Anthropic 的建议本质上是在推动一种 `eval-driven development`：先用评估定义能力，再推动 agent 朝这个方向演进。

## 简要结论

如果把这篇文章压缩成一句话，它想表达的是：`agent 的评估不是给结果打个分，而是构建一套能稳定衡量真实任务完成质量的工程系统。` 真正有用的 eval，不只是能测出模型强弱，更能帮助团队尽早定义成功标准、系统发现回退、解释模型升级收益，并把 agent 开发从“凭感觉调系统”推进到“围绕可验证目标迭代系统”。
