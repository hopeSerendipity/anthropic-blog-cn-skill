# Building agents with the Claude Agent SDK - 笔记

## 文章主旨

这篇文章主要想表达的是，Anthropic 发现原本为编程场景打造的 `Claude Code SDK`，其实并不只适用于写代码，它同样适合研究、客服、个人助理、信息整理等大量非编程任务，因此将其更名为 `Claude Agent SDK`。文章真正想强调的不是一次产品命名调整，而是一种更普遍的 agent 构建思路：如果想让模型真正完成工作，就不能只让它“回答”，而要让它能够像人一样使用电脑，在收集信息、执行操作、检查结果和反复迭代的循环中完成任务。

## 关于 Agent 构建的核心观点

### 1. Agent 的核心不只是 prompt，而是完整的工作闭环

文章认为，一个真正有用的 agent 不能只靠一段提示词去“尽量一次答对”，而应该具备完整的执行循环，也就是先收集上下文，再采取行动，然后验证结果，最后根据反馈继续迭代。这个观点的重要性在于，它把 agent 的能力来源从“模型本身有多聪明”转移到了“系统是否允许它持续修正自己”上。换句话说，好的 agent 不是一开始就完美，而是在循环中逐步逼近正确结果。

### 2. “给 agent 一台电脑”是最关键的设计原则

文章反复强调，Claude 之所以能在编码之外处理研究、文档、可视化等任务，关键不是因为模型突然懂了更多领域知识，而是因为它被赋予了使用电脑的能力。这里的“电脑”并不只是一个比喻，而是指终端、文件系统、脚本执行、搜索和编辑等实际操作环境。这个观点非常有价值，因为它提醒我们，很多现实任务本质上不是纯语言问题，而是“查资料、运行工具、改文件、验证结果”的操作问题，agent 的能力上限往往取决于它能否真正进入这个工作环境。

### 3. 文件系统本身就是上下文设计的一部分

文章提出了一个很值得记住的判断：文件夹结构和文件组织方式本身，就是上下文工程的一部分。如果历史对话、日志、用户资料、任务记录都被清晰地放在合适的位置，agent 就能像人一样去查找、筛选和组合这些信息，而不是被动等待你把所有内容塞进提示词里。这个观点的意义在于，它把“怎么存信息”也纳入了 agent 设计范畴，因为对于 agent 来说，信息是否可发现，和信息本身是否存在一样重要。

### 4. 应优先使用 agentic search，而不是过早依赖 semantic search

文章建议，构建 agent 时应该优先让它通过文件搜索、脚本、命令行等方式主动查找信息，而不是一开始就上向量检索或语义搜索。因为 agentic search 更透明、更容易理解其过程，也更容易维护和调试；而 semantic search 虽然可能更快，但通常更黑盒、维护成本更高，而且在复杂场景下未必更准确。这个观点的底层逻辑是，先让 agent 以一种人类可以理解和审计的方式工作，再在确定瓶颈之后用更复杂的检索机制去补速度，而不是一开始就把系统搭成难以解释的形态。

### 5. Subagent 的价值不仅是并行，更是上下文隔离

文章提到子 agent 时，不只是把它当作“多线程处理任务”的手段，而是强调了它的另一个核心作用，即隔离上下文。每个子 agent 可以在自己的上下文窗口中独立搜索、筛选和处理信息，最后只把真正有用的结果返回给主 agent，而不必把完整过程全部带回来。这个设计很有现实意义，因为很多任务的问题不在于“模型不会做”，而在于主上下文里塞了太多无关内容，导致判断质量下降。subagent 因此不仅提升效率，也是一种控制上下文污染的方式。

### 6. 长时间运行的 agent，必须主动管理上下文

文章提到 `compact` 功能时，其实是在强调一个经常被忽视的问题：agent 一旦运行时间变长，上下文就会逐渐膨胀，而上下文管理本身会成为系统稳定性的关键部分。如果没有压缩、总结和重组机制，早期信息会越来越难被利用，后续推理也会越来越混乱。这个观点的价值在于，它提醒我们，agent 设计不是只关心“会不会调用工具”，还必须关心“长时间工作后是否还能维持清晰状态”，否则系统很容易在后半程失控。

### 7. 工具设计会直接决定 agent 的行为质量

文章认为，工具是 agent 在执行时最优先考虑的动作，因此工具设计不能随意堆砌，而要尽量清晰、稳定、贴近真实高频任务。因为工具一旦定义得含糊、冗余或边界不清，agent 在调用时就更容易误判，最终表现出不稳定、低效率甚至重复试错的行为。这个观点很重要，因为它说明 agent 的表现并不完全取决于模型能力，工具的命名、职责划分和输入输出结构，本身就是行为约束的一部分，好的工具设计实际上是在帮模型更容易做对事。

### 8. Bash 和脚本不是补充能力，而是通用执行能力

文章把 Bash 和脚本放在一个很高的位置，原因在于它们能让 agent 处理大量无法提前做成专门工具的任务。现实工作中经常会遇到一些临时性的、细碎的、非标准化的问题，比如清洗一个 CSV、提取一个 PDF 的文字、统计一段日志或快速改写一批文件，这些都不适合每次都专门开发新接口，而脚本正好提供了即时、灵活、低成本的解决方式。这个观点说明，强 agent 不一定要预置一切能力，更实用的做法往往是让它拥有“现写小程序解决问题”的自由度。

### 9. 代码生成不仅是输出能力，也是执行能力

文章特别强调代码生成的价值，不只是因为 Claude 会写代码，而是因为代码本身是一种非常适合 agent 使用的执行媒介。代码精确、可组合、可重复调用，因此对于复杂任务来说，用代码表达一个操作过程，往往比让模型直接用自然语言一次性完成更稳定。比如生成 Excel、PPT 或 Word，本质上都不是“写一段描述”，而是“先把操作逻辑变成代码，再让代码稳定地产生结果”。这个观点的核心意义在于，agent 不应只把代码看作最终产物，而应把它看作完成复杂工作的中间层。

### 10. MCP 的价值在于让 agent 更容易进入真实业务环境

文章提到 MCP 时，重点不是协议本身，而是它解决了 agent 接入外部系统时最麻烦的那部分工作，包括认证、接口调用和集成标准化。对于 agent 来说，真正产生业务价值往往不是在一个封闭环境里推理，而是能够进入 Slack、GitHub、Google Drive、Asana 这类真实工具链中去获取信息和执行动作。MCP 的意义就在于降低这类连接成本，让开发者不必每次都重复造集成层。这个观点很有价值，因为它说明“连接现实系统的能力”是 agent 实用性的关键组成部分。

### 11. 一个可靠的 agent 必须具备验证自己的能力

文章认为，agent 是否值得信任，很大程度上取决于它是否能检查自己的输出并根据结果修正自己。作者给出的三类方式中，最重要的是规则验证，因为它最明确、最稳定，也最容易自动化，比如 lint、格式检查、字段合法性检查和类型检查；视觉验证适合界面、网页和 HTML 邮件这类任务，因为最终效果本来就要靠视觉来判断；至于 LLM as a judge，虽然有时能提供补充帮助，但通常不如规则反馈稳健。这个观点的核心不在于“多做一次检查”，而在于必须给 agent 可执行、可判断、可反馈的标准，否则所谓自我修正只会停留在口头层面。

### 12. 优化 agent 的最好入口是失败案例，而不是成功案例

文章最后提出的改进方法非常务实，即与其反复欣赏 agent 成功完成任务的过程，不如重点研究它失败的时候到底缺了什么。失败可能来自上下文不足、搜索入口设计不合理、工具不够好、规则不够明确，或者新增功能后整体行为开始波动。这个观点背后其实是在说明，agent 调优并不是神秘的“炼丹”，而是围绕信息获取、执行能力和反馈机制不断补短板的过程。只有认真看失败样本，才能真正知道问题出在模型、工具、上下文还是评估方式上。

## 简要结论

如果把整篇文章压缩成一句话，它想表达的是：`高效 agent 不是一个会聊天的模型，而是一个能获取上下文、使用工具、验证结果并持续迭代的工作系统。` 它真正有价值的地方，不在于介绍某个 SDK 的新名字，而在于总结了一套很实用的 agent 构建观：先让 agent 像人一样工作，再逐步优化它查信息、做事情和检查结果的方式。
