# 深入 OpenClaw：一个持久型 AI Agent 实际如何运作 - 笔记

## 文章主旨

这篇文章解释了 OpenClaw 这类持久型 AI Agent 的实际架构：它不是一个普通聊天网页，而是运行在用户机器上的本地进程，通过 Gateway 接入 WhatsApp、Telegram、Slack 等消息平台，再用持久文件记忆、工具技能和定时 heartbeat 形成可以长期在线、主动检查和执行任务的 Agent。文章的核心判断是：OpenClaw 的技术并不神秘，关键价值在于“本地、持久、可检查、可扩展”，但这种能力也带来真实的安全风险。

## 核心观点

### Gateway 是整个系统的中枢

OpenClaw 通过一个默认监听 `127.0.0.1:18789` 的 Node.js Gateway 统一管理所有平台连接。所有外部消息先进入 Gateway，再经过认证、路由、安全检查和会话加载，最后才交给 Agent 处理。这个中心化设计尤其适合处理 WhatsApp Web 这类只允许单活会话的平台，因为 Gateway 可以作为唯一平台会话，再在内部管理多个 Agent 对话。

### 默认本地绑定降低了意外暴露风险

Gateway 默认绑定到 `127.0.0.1`，意味着只有本机进程能访问它，外部网络无法直接连入。考虑到 Agent 可能具备文件系统、终端和 API 权限，这个默认值非常关键：OpenClaw 的安全边界首先来自“默认不公开”，只有用户主动改成对外绑定时才会暴露到网络。

### Agent 循环把消息变成带上下文的行动

消息到达后不会被简单转发给 LLM，而是进入一个 Agent 循环：根据发送者和平台选择会话，从文件系统加载历史上下文，把消息、工具和技能一起交给模型，让模型决定是否调用工具并生成回复。这让 Agent 能把“查一下日历”变成真正的 API 调用，也能在不同平台上保持同一套上下文。

### 持久记忆来自普通文件而不是数据库

OpenClaw 把行为、人格、工具、技能和记忆都存成 `~/clawd/` 下的 Markdown 文件，例如 `AGENTS.md`、`SOUL.md`、`TOOLS.md`、`SKILL.md` 和带时间戳的对话日志。这个设计牺牲了一些托管便利性，但换来了透明性、可编辑性和可版本控制：用户可以直接打开文件查看 Agent 知道什么，也可以用 Git 追踪和回滚变化。

### Heartbeat 让 Agent 具备主动性

OpenClaw 可以通过 cron 按固定间隔唤醒 Agent，读取 `HEARTBEAT.md` 并判断是否需要通知用户。为了避免成本失控，它先运行廉价的确定性检查，例如邮件、日历或系统告警，只有出现重要信号时才调用 LLM 做理解和决策。这让它可以主动报告服务器故障、重要邮件或市场变化，而不是等用户提问。

### 技能系统用 Markdown 扩展能力

OpenClaw 的技能不是编译插件，而是 `~/clawd/skills/<skill-name>/SKILL.md` 里的运行时说明。Agent 读取这些技能文件后，就知道如何调用 API 或执行工作流。安装新技能不需要重启或重新编译，这让能力扩展变得轻量，也与整个系统“文件即配置、文件即记忆”的设计保持一致。

### 执行位置决定数据与控制权归属

OpenClaw 的技能和工具在进程所在的位置执行，可以是本地机器、VPS 或托管容器。相比云端 AI 工具把数据和执行放在供应商基础设施上，OpenClaw 更强调由用户掌控运行环境、数据存储和 API 调用路径。代价是用户也必须承担部署、权限配置和系统安全管理。

### 安全模型默认假设 LLM 会被欺骗

文章强调 OpenClaw 不能在 LLM 层面彻底防止提示注入，所以需要在 Gateway 和工具层建立多层限制：允许列表、设备令牌权限、读写权限分离，以及危险操作审批。这个设计并不能消灭攻击，但可以限制单次被攻破对话的破坏范围，让用户有机会发现并介入。

### OpenClaw 的价值在组合，而不是单点技术突破

文章最后指出，OpenClaw 有趣之处不在于某个革命性算法，而在于把持久执行、主动 heartbeat、多平台消息接入、文件化记忆、Markdown 技能和本地运行组合成一个可检查、可修改的系统。它适合需要高控制权和可扩展性的场景，但不适合不了解安全权衡就把系统完全交给 AI 的使用方式。
