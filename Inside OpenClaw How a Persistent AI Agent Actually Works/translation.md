# 深入 OpenClaw：一个持久型 AI Agent 实际如何运作

Aamitesh Anand  
2026 年 2 月 19 日  
Insights

## 引言

OpenClaw 最初叫 ClawdBot，如今到处都在流行。人们正在用它构建一些非常夸张的东西：一个可以通过 Telegram 重建整个网站的 AI Agent；一个人类只作为访客存在的 AI Agent 平台；以及把系统的完整访问权交给一个 AI，结果它可能因为一次提示注入攻击误删 6000 封邮件。

和坐在网页界面背后的 ChatGPT 或 Claude 不同，OpenClaw 会作为一个持久进程运行在你的硬件上。你可以通过 WhatsApp、Telegram 或 Slack 给它发消息。它会回复你。它可以在你睡觉时检查事情。它可以访问你的文件系统、终端，以及你交给它的任何 API。

它的可能性非常狂野，安全风险也是真实存在的。理解它背后的技术架构，就能同时解释这两点，而且它比你想象的更简单。我们来看看它实际是如何运作的。

## Gateway 架构：中枢神经系统

OpenClaw 会在你的机器上以单个 Node.js 进程运行，默认监听 `127.0.0.1:18789`。这个进程叫作 `Gateway`，它会同时管理所有消息平台连接：WhatsApp、Telegram、Discord、Slack、Signal 等。

可以把它理解为中枢神经系统。任何平台发来的每一条消息都会经过 Gateway。你的 Agent 生成的每一条回复也会通过它发回去。所有通信都通过 `WebSocket` 协议完成，这种协议会保持连接打开，并允许实时的双向消息传递。

## 会话状态、路由与安全

Gateway 负责三个关键功能：

- 会话状态管理；
- 消息路由；
- 安全执行。

当一条来自 WhatsApp 的消息抵达时，Gateway 会根据用户、对话上下文或你配置的路由规则，判断应该由哪个 Agent 会话来处理。它会加载合适的会话状态，把消息传给 Agent，等待 LLM 生成回复，然后再把回复通过正确的平台连接路由回去。

这种中心化设计解决了一个真实的技术问题。WhatsApp Web 同一时间只允许一个活跃会话。如果你尝试运行多个实例，它们会互相冲突，并把对方踢下线。Gateway 就扮演那个唯一会话的角色，然后在内部管理多个 Agent 对话。你只需要配置一次 WhatsApp，后续的所有事情都由 Gateway 处理。相同原则也适用于其他平台。

## 连接与认证流程

当某个平台想要连接时，它会建立一个 WebSocket 连接，并发送一个带有设备身份的连接请求。意思大致是：“我是运行在设备 XYZ 上的 WhatsApp，我想和你的 Agent 通信。”Gateway 会检查自己的配对存储。如果这个设备之前从未连接过，它会拒绝连接，并等待明确批准。

批准之后，Gateway 会发放一个作用域限定在特定权限内的设备令牌。这个令牌决定了该设备可以做什么：

- 它可以代表哪些用户发消息；
- 它可以访问哪些 Agent 会话；
- 它拥有哪些能力。

之后的连接会使用这个令牌进行认证，而不是每次都要求重新批准。

## 认证后的消息路由

某个平台完成认证之后，它发送的每一条消息都会经过路由逻辑。Gateway 会根据你配置的规则，决定消息应该去哪里，以及 Agent 是否应该响应：

- 来自允许列表中用户的消息会被处理；
- 来自未知用户的消息会在 Agent 看到之前被丢弃；
- 私信会路由到你的个人助理 Agent；
- 群聊可能只有在有人直接 `@` Agent 时才触发回复。

## 网络绑定：默认只在本地开放

所有这些路由和认证都发生在你的本地机器上。Gateway 默认绑定到 `127.0.0.1`，也就是 `localhost`，而不是 `0.0.0.0`，也就是所有网络接口。这个网络绑定决定了谁一开始就能连接到 Gateway。

绑定到 `127.0.0.1` 意味着只有运行在你机器上的进程能访问 Gateway，外部网络无法访问。除非你有意重新配置绑定方式，否则你的 Agent 不会从机器外部暴露出去。考虑到 Gateway 可以访问你的文件系统、终端和已连接的 API，这一点对避免意外公开暴露至关重要。

每条消息都遵循同一条路径：

```text
平台 -> Gateway 认证 -> 路由逻辑 -> Agent 会话加载 -> LLM 处理 -> 回复生成 -> Gateway -> 平台投递
```

一个进程。所有平台。集中控制。而且除非你明确决定改变配置，否则一切都留在本地。

现在我们理解了消息如何抵达 Agent，接下来看看消息抵达后会发生什么。

## Agent 循环：从消息到行动

当消息到达 Gateway 时，它不会被盲目转发给 LLM。这里有一个处理循环，会把你说的“检查我的日历”变成带有上下文的真实回复。

Gateway 会根据消息是谁发来的、来自哪里，将消息路由到对应的 Agent 会话。该会话会从文件系统加载对话历史，也就是你过去对这个 Agent 说过的一切，不只是当前对话，还包括之前的对话。这就是为什么你的 Agent 会记得你上周二问过某个项目。

Agent 会把消息连同可用工具和技能一起传给 LLM。模型处理请求，判断是否需要调用工具，比如检查日历或发送邮件，然后执行这些动作并生成回复。回复会通过 Gateway 流回你发消息的平台。

## 持久存在的上下文

和每次都是全新对话的 ChatGPT 不同，OpenClaw 的会话不会重置。Agent 知道你是谁，知道你之前问过什么，也知道你的工作区里有什么。如果你上周告诉它你正在做 Project XYZ，它会记得。如果你在工作区里保存了笔记，它也可以引用。

这种持久性之所以成立，是因为所有内容都保存在你机器上的文件里。Agent 每次处理消息时都会重新加载上下文，但关闭聊天并不会让这些上下文消失。你也不会被锁定在一个 LLM 上：可以为复杂推理配置 Claude，为创意任务配置 GPT-4，或者为简单查询配置更便宜的模型。无论使用哪个模型，Agent 循环的工作方式都一样。

这种基于文件的记忆方式让持久性成为可能。接下来看看它实际是怎样工作的。

## 持久记忆：一切都是文件

OpenClaw 不使用数据库。所有内容都以 Markdown 文件的形式存储在 `~/clawd/` 中。

Agent 的行为定义在 `AGENTS.md` 里。它的个性和核心指令存储在 `SOUL.md` 里。可用工具列在 `TOOLS.md` 里。你安装的技能保存在 `~/clawd/skills/<skill>/SKILL.md` 中。记忆日志则是带时间戳的文件，文件名类似 `2026-02-10-conversation.md`。

打开任何文本编辑器，你就能看到 Agent 到底知道什么。想检查它还记得上次项目讨论的哪些内容？打开记忆日志。想修改它如何回应日历请求？编辑 `AGENTS.md`。想看看它有哪些工具权限？阅读 `TOOLS.md`。

因为一切都是纯文本，版本控制无需额外设置就能工作。在 `~/clawd/` 中运行 `git init`，每一次变化都会被追踪。你可以看到什么时候添加了新技能，什么时候 Agent 更新了长期记忆，或者什么时候你修改了它的核心指令。如果出了问题，可以回滚到之前的提交。备份也很简单，复制整个目录即可。

## 记忆如何组织自己

OpenClaw 将记忆分成多个层次。每日日志捕获短期上下文：你今天聊了什么、哪些任务正在进行、分享了哪些链接。这些带时间戳的文件会随着时间累积。

长期记忆由 Agent 自己整理。随着对话发生，Agent 会判断哪些信息重要到值得永久记住。也许你告诉它自己偏好简洁回复。也许你给了它关于如何处理某类请求的长期指令。这些信息会被写入长期记忆文件，并跨会话持久存在。

如果你正在分析一个数据集，而电脑崩溃了，Agent 会在重新启动后加载工作区状态。它知道你停在了哪里，因为这些状态保存在它重启后能读取的文件里。

不过，仅有记忆还不能让 Agent 主动行动。要做到这一点，OpenClaw 需要一种机制，让它能在你没有提问时醒来检查事情。这就是 heartbeat 的作用。

## Heartbeat：主动型 Agent

大多数 AI 助手都会等你先提出问题。OpenClaw 不一定需要这样。

一个 cron 任务会按你配置的间隔唤醒 Agent，默认是每 30 分钟一次。Agent 会检查 `HEARTBEAT.md` 中的指令，运行一次推理循环，并判断是否需要告诉你某件事。你不需要主动发送提示。

这就是你获得主动通知的方式。你的服务器在凌晨 3 点宕机，Agent 会通过 Telegram 给你发消息。你监控的一只股票下跌 15%，它执行卖出订单，并通过 WhatsApp 确认。客户发来了三封紧急邮件，它会立刻标记出来，而不是等你自己去检查。

## 先做低成本检查

OpenClaw 并不会在每次 heartbeat 时都调用 LLM。那会很快烧掉 API 成本。相反，它采用两层方法：先做低成本检查，只有需要时才调用模型。

Agent 会先运行快速、确定性的脚本，检查新邮件、日历变化或系统告警。这些只是简单的模式匹配或 API 查询，不产生成本。只有当某个重要变化出现时，Agent 才会升级到 LLM 来解释并做决策。

举个例子，低成本检查发现“房东发来一封新邮件”。这是一个信号。Agent 随后调用 Claude 或 GPT-4 阅读邮件，结合之前关于租约的对话上下文理解内容，然后决定是否需要通知你或采取行动。如果 heartbeat 没发现新情况，就不会发生 LLM 调用。

这种设计在保持响应能力的同时让成本保持合理。你不会在没有重要事情发生时，为每天 48 次 LLM 调用付费。

## 配置

`HEARTBEAT.md` 的配置如下：

```text
every: "30m"
target: "whatsapp:+1234567890"
active_hours: "9am-10pm"
```

`active_hours` 设置可以避免 Agent 在凌晨 2 点因为非紧急更新把你叫醒。

`target` 指定 heartbeat 消息要发送到哪个平台和联系人。

`every` 参数控制频率。标准监控可以设为 `"1h"`；如果你正在积极工作，需要更密集的检查，可以设为 `"15m"`；如果你需要接近实时的响应，可以设为 `"5m"`。

每次 heartbeat 周期都会加载 Agent 当前上下文，检查 heartbeat 指令中定义的条件，并且只有在确实有事情需要注意时才发送消息。它不是每 30 分钟骚扰你一次，而是每 30 分钟检查一次，并在有理由时开口。

这种主动能力是内置的，但你可以扩展 Agent 在这些检查期间实际能做什么。这就是技能的作用。

## 技能与执行：扩展 Agent 能力

OpenClaw 使用基于技能的架构，能力定义在 Markdown 文件中，而不是编译后的代码里。

每个技能都以 `~/clawd/skills/<skill-name>/SKILL.md` 的形式存在，其中包含与 API 交互或执行工作流的指令。Agent 会在运行时读取这些文件，以理解可用能力。安装会立即生效，无需重新编译或重启服务器。ClawHub 上已经有超过 100 个社区技能，覆盖 Gmail、浏览器自动化、家庭控制等场景。

## 执行模型

技能会在 OpenClaw 进程所在的位置执行，可以是你的本地机器、VPS，也可以是托管容器。架构保持一致：Gateway 路由消息，Agent 从文件系统加载技能，LLM 调用直接发生，不通过某个供应商代理，结果再写回本地存储。

| 方面 | 云端 AI 工具 | OpenClaw |
| --- | --- | --- |
| 数据存储 | 供应商服务器 | 进程运行的位置 |
| 执行 | 供应商基础设施 | 你的硬件或 VPS |
| API 调用 | 通过供应商代理 | 由 Agent 直接调用 |

工具限制会在 Gateway 层面设置边界。你可以在沙盒模式下运行 Agent，也就是为了安全限制能力；也可以在完全访问模式下运行，让它拥有不受限制的系统控制权。在沙盒模式下，它会阻止写入文件系统和 shell 访问。完全访问模式允许你使用终端命令并控制浏览器。如果 LLM 试图做某件不被允许的事，Gateway 会在事情发生前拦住它。

无论进程运行在哪里，连接到消息平台都需要认证和安全执行。这正是 Gateway 的关键作用所在。

## 安全与多平台处理

Gateway 会在路由层执行安全策略，而不仅仅是在连接时做一次检查。

一旦平台完成认证并连接，每一条消息都会在抵达 Agent 之前经过安全检查。允许列表控制哪些用户或群组能得到响应。如果不在列表中的人发来消息，Gateway 会在 Agent 看到之前直接丢弃。这个机制适用于所有平台：WhatsApp、Telegram、Discord、Slack，使用同一套允许列表配置。

## 多平台路由如何工作

Channel Layer 位于平台连接与 Agent 之间。WhatsApp 消息是一种格式，Telegram 是另一种格式，Discord 又是第三种格式。Channel Layer 会把这些消息适配为一种通用内部结构，这样 Agent 就不需要编写平台专用代码。它还会处理平台事件，例如表情回应、正在输入提示和已读回执。

这种抽象意味着你可以编写一条适用于所有平台的路由规则。“只在群聊中被 `@` 时回复”这条规则，无论消息来自 Slack 还是 Discord，都能以同样方式工作。Channel Layer 会把平台特有的 mention 格式转换成 Gateway 能理解的标准结构。

## 安全架构：分层限制

OpenClaw 的架构假设 LLM 可能被欺骗。提示注入攻击是真实存在的，架构无法在 LLM 层面阻止它们，因此会通过多层执行限制来降低损害：

- 工具审批工作流会把危险操作，例如删除文件、shell 命令、支付，放到明确的用户确认之后；
- 作用域权限会区分读写访问，例如读取邮件与发送邮件、查询数据库与修改数据库；
- 设备令牌能力会限制每个已连接设备能做什么，例如只能私信、不能进入群聊、只读模式。

一次被攻破的对话不应该获得一切访问权。这些层并不能阻止一个掌控 LLM 所读内容的坚定攻击者，但它们能拖慢攻击者，让你有足够时间注意到并介入。引言中提到的 6000 封邮件删除事件并不是一个设计缺陷，而是展示了为什么这些限制很重要，以及为什么让 AI Agent 拥有完整系统访问权需要理解相关风险。

这个架构把控制权交给你：

- 选择哪些平台可以连接；
- 选择哪些用户会得到响应；
- 选择哪些工具可用；
- 选择哪些操作需要审批。

这份控制权，就是运行一个能访问你系统的持久型 Agent 所需要付出的权衡。

## 还有什么？

OpenClaw 的架构出人意料地简单：Gateway 负责路由消息，Agent 循环用 LLM 和工具处理消息，记忆以文件形式持久化，技能扩展能力，heartbeat 负责主动检查。没有数据库，没有微服务，也没有供应商锁定。

它的设计选择，包括基于文件的记忆、Markdown 技能、本地执行、默认假设会被攻破，都把透明性和控制权置于便利性之上。你能确切看到 Agent 知道什么、能做什么、运行在哪里。代价是你需要管理基础设施，并接受把系统访问权交给 AI 所带来的安全风险。

OpenClaw 有趣的地方并不在于革命性的技术，而在于它把持久执行、主动行为、多平台集成和模块化能力组合进了一个你可以检查和修改的架构里。它是否值得运行，取决于你正在构建什么，以及你需要多少控制权。
