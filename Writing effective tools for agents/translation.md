# Writing effective tools for agents - 中文翻译

原文标题：`Writing effective tools for agents — with agents`

Agent 的效果，很大程度上取决于我们给它们什么样的工具。本文分享了如何编写高质量的工具与评测，以及如何通过让 Claude 为自己优化工具，进一步提升整体表现。

`Model Context Protocol`（`MCP`）可以让 LLM agent 使用多达数百种工具来解决真实世界任务。但问题是，我们该如何让这些工具尽可能高效？

在这篇文章中，我们介绍了一些最有效的方法，用来提升各种 agentic AI 系统的表现。<sup>1</sup>

一开始，我们会讲你可以如何：

- 构建并测试工具原型
- 使用 agent 为你的工具建立并运行完整评测
- 与 Claude Code 这样的 agent 协作，自动提升工具表现

最后，我们会总结一路实践中得到的一些高质量工具编写原则：

- 选择应该实现哪些工具，以及不该实现哪些工具
- 通过工具命名空间划定清晰的功能边界
- 让工具返回对 agent 真正有意义的上下文
- 为 token 效率优化工具响应
- 对工具描述和规格进行 prompt engineering

构建一套评测体系，可以让你系统性地衡量工具表现。你也可以借助 Claude Code，围绕这套评测自动优化你的工具。

## 什么是工具？

在计算领域，确定性系统在相同输入下每次都会产生相同输出，而非确定性系统，比如 agent，即使起始条件相同，也可能给出不同响应。

在传统软件开发中，我们建立的是确定性系统之间的契约。比如 `getWeather("NYC")` 这样的函数调用，每次都会以完全相同的方式获取纽约的天气。

工具则是一类新的软件，它体现的是“确定性系统”和“非确定性 agent”之间的契约。当用户问“我今天要不要带伞？”时，agent 可能会调用天气工具，也可能凭常识直接回答，甚至可能先追问用户所在位置。有时，agent 还可能产生幻觉，或者根本没有理解该如何使用工具。

这意味着，我们在为 agent 编写软件时，需要从根本上改变思路：我们不能再像为其他开发者或系统编写函数和 API 那样去编写工具和 MCP server，而是必须以 agent 的使用方式为中心来设计它们。

我们的目标，是通过工具扩大 agent 能够高效解决任务的范围，让它们可以借助多种成功策略完成广泛的真实工作。幸运的是，根据我们的经验，那些对 agent 来说最“顺手”的工具，通常对人类来说也同样直观易懂。

## 如何编写工具

这一部分会介绍，如何与 agent 协作，既让它帮助你编写工具，也让它帮助你改进工具。首先，快速搭一个工具原型并在本地测试。接着，运行一套完整评测来衡量后续修改带来的影响。然后，你可以持续与 agent 一起反复做“评测 -> 改进”的循环，直到你的工具在真实任务上取得足够强的表现。

### 构建原型

如果不亲自上手，往往很难提前判断哪些工具对 agent 来说顺手，哪些不顺手。因此，第一步应该是尽快搭一个工具原型。

如果你使用 Claude Code 来编写工具，甚至可能一次性生成初版，那么最好把这些工具所依赖的软件库、API 或 SDK 文档提供给 Claude，包括 MCP SDK 在内。适合 LLM 阅读的文档通常可以在官方文档站点里的扁平 `llms.txt` 文件中找到。

如果你把工具封装进一个本地 MCP server 或一个 Desktop extension（`DXT`），就可以在 Claude Code 或 Claude Desktop 应用中连接并测试这些工具。

要把本地 MCP server 接入 Claude Code，可以运行：

```bash
claude mcp add <name> <command> [args...]
```

要把本地 MCP server 或 DXT 接入 Claude Desktop，可以分别进入 `Settings > Developer` 或 `Settings > Extensions`。

你也可以把工具直接传入 Anthropic API 调用中，以编程方式进行测试。

要亲自试用这些工具，找出它们的粗糙之处。同时，向你的用户收集反馈，逐步形成对这些工具实际适用场景以及可能触发它们的 prompt 方式的直觉。

### 运行评测

接下来，你需要通过评测来衡量 Claude 使用这些工具的效果。先生成大量基于真实使用场景的评测任务。我们建议你与一个 agent 协作，让它帮助分析结果，并判断应如何改进工具。

#### 生成评测任务

有了早期原型之后，Claude Code 可以很快探索你的工具，并生成几十组 prompt 与响应配对。prompt 应该来源于真实世界的使用方式，并建立在真实的数据源和服务之上，例如内部知识库和微服务。我们建议避免那种过于简单、过于表面的“沙盒环境”，因为这类环境无法用足够的复杂度来真正检验你的工具。强评测任务往往需要多次工具调用，甚至可能多达几十次。

下面是一些较强的任务示例：

- 下周安排一次与 Jane 的会议，讨论我们最新的 Acme Corp 项目。附上上次项目规划会议的笔记，并预订一间会议室。
- 客户 ID `9182` 报告称，他们在一次购买尝试中被重复扣费三次。请找出所有相关日志，并判断是否有其他客户也受到了同样问题的影响。
- 客户 Sarah Chen 刚刚提交了一次取消请求。请准备一个挽留方案，并确定：`(1)` 她为什么想离开，`(2)` 哪种挽留方案最有吸引力，`(3)` 在提出方案前有哪些风险因素需要注意。

下面则是一些较弱的任务示例：

- 下周给 `jane@acme.corp` 安排一个会议。
- 在支付日志中搜索 `purchase_complete` 和 `customer_id=9182`。
- 通过客户 ID `45892` 找到取消请求。

每条评测 prompt 都应该对应一个可验证的响应或结果。验证器既可以很简单，比如把标准答案与采样结果做精确字符串比对；也可以更复杂，比如让 Claude 来判断响应是否正确。要避免验证器过于严苛，因为格式、标点或合理的不同表达方式等表面差异，可能会让本来正确的答案被误判为错误。

对于每一组 prompt-响应配对，你也可以额外指定“你期望 agent 调用了哪些工具”，以衡量 agent 是否真正理解了各个工具的用途。不过，由于很多任务本来就可能有多条有效解法，所以应尽量避免把策略规定得过死，也要避免针对单一路径过拟合。

#### 运行评测

我们建议你用直接的 LLM API 调用，以编程方式运行评测。可以使用简单的 agent 循环，也就是一个包裹着“LLM API 调用”和“工具调用”交替执行的 `while` 循环：每个评测任务对应一个循环。每个评测 agent 都只接收一条任务 prompt 和同一组工具。

在评测 agent 的 system prompt 中，我们建议让 agent 不只输出结构化响应块用于验证，还输出 reasoning 和 feedback 块。如果这些内容出现在工具调用块和响应块之前，可能会通过触发链式思维（`CoT`）行为，提升 LLM 的有效智能水平。

如果你用的是 Claude 来跑评测，也可以直接打开 `interleaved thinking`，获得类似能力。这能帮助你更清楚地观察 agent 为什么会或不会调用某些工具，并暴露工具描述和规格中值得改进的地方。

除了顶层准确率之外，我们还建议收集其他指标，例如单次工具调用和整个任务的总耗时、工具调用总次数、总 token 消耗以及工具报错情况。跟踪工具调用路径，有助于你看出 agent 常走的工作流，也能发现是否有一些工具可以进一步整合。

#### 分析结果

Agent 是发现问题和提供反馈的好帮手，它们可以指出从工具描述相互矛盾，到工具实现低效，再到工具 schema 令人困惑等各种问题。不过也要记住，agent 在反馈和响应中“没有说出来的部分”，往往和它说出来的内容一样重要。LLM 并不总是“怎么想就怎么说”。

观察你的 agent 在哪里卡住、哪里困惑。阅读评测 agent 的 reasoning 和 feedback（或 CoT），找出粗糙边缘；再回看原始 transcript，包括工具调用和工具响应，以捕捉那些没有在 CoT 中明确说出的行为。你需要学会“读空气”，因为评测 agent 自己并不一定知道正确答案，也未必知道最佳策略。

同时，分析工具调用指标。如果出现大量重复工具调用，可能说明你的分页参数或 token 限制参数需要重新调整；如果大量报错来自无效参数，则说明工具可能需要更清晰的描述或更好的示例。比如，在我们发布 Claude 的网页搜索工具时，就发现 Claude 会无意义地把 `2025` 拼到工具的 `query` 参数里，导致搜索结果产生偏差并降低表现。后来，我们通过改进工具描述，把 Claude 引导回了正确方向。

### 与 agent 协作改进工具

你甚至可以让 agent 来帮你分析结果并改进工具。只需要把评测 agent 的 transcript 拼接起来，然后粘贴进 Claude Code。Claude 很擅长分析 transcript，并同时重构大量工具，例如确保当你添加新改动时，工具实现和工具描述仍然保持一致。

事实上，本文中的大部分建议，都是我们用 Claude Code 反复优化内部工具实现后总结出来的。我们的评测建立在内部工作空间之上，复杂度与真实内部工作流一致，包括真实项目、文档和消息。

我们还依赖保留测试集，确保没有对“训练用评测集”过拟合。这些测试集显示，即便我们已经有了所谓“专家级”的工具实现，无论这些工具是研究人员手写的，还是 Claude 自己生成的，我们仍然可以进一步榨出更多性能提升。

接下来，我们会分享从这一过程中提炼出的经验原则。

## 编写高效工具的原则

这一部分将把我们学到的经验提炼成几条指导原则。

### 为 agent 选择合适的工具

工具并不是越多越好。我们经常看到的一个错误，是有人只是简单把现有软件功能或 API endpoint 包装成工具，却没有思考这些工具是否真的适合 agent。这背后的原因在于，agent 与传统软件有不同的“可供性”，也就是说，它们对“自己可以采取哪些行动”的感知方式并不一样。

LLM agent 的上下文是有限的，也就是它一次能处理的信息量有限，而计算机内存则便宜且充裕。以在通讯录中查找联系人为例，传统软件可以很高效地逐个处理联系人列表，一项一项比对，再继续向下处理。

但如果一个 LLM agent 调用的工具一次性返回了所有联系人，它就不得不逐 token 去阅读每一项，这会把宝贵的上下文浪费在大量无关信息上。这就像在纸质通讯录里从第一页开始一页页硬翻，属于暴力搜索。对 agent 和人类来说，更自然的做法都是先跳到更可能相关的页面，例如按字母顺序缩小范围。

我们建议先构建少量、经过认真思考、面向高影响工作流的工具，并让它们与你的评测任务对齐，然后再逐步扩展。以通讯录为例，与其实现一个 `list_contacts`，不如实现一个 `search_contacts` 或 `message_contact`。

工具也可以整合多个能力，把原本分散的若干操作或 API 调用收敛到一次工具调用里。例如，工具可以在响应中附带相关元数据，或者把那些经常需要串联的多步骤任务合并成一步完成。

例如：

- 与其实现 `list_users`、`list_events` 和 `create_event`，不如实现一个 `schedule_event`，让它自动查找空闲时间并安排会议。
- 与其实现 `read_logs`，不如实现一个 `search_logs`，只返回相关日志行以及必要的上下文。
- 与其分别实现 `get_customer_by_id`、`list_transactions` 和 `list_notes`，不如实现一个 `get_customer_context`，一次性汇总该客户近期最相关的信息。

你构建的每个工具都应该拥有清晰且独立的用途。工具应当帮助 agent 像人一样，把任务拆解并解决；同时，它们还应减少那些原本会由中间输出占据的上下文消耗。

工具太多，或者工具之间高度重叠，也会分散 agent 的注意力，让它更难采取高效策略。因此，谨慎而有选择地规划“该做哪些工具、哪些不要做”，往往会带来很大回报。

### 给工具做命名空间划分

你的 AI agent 很可能会接入几十个 MCP server，以及数百种不同工具，其中还包括其他开发者编写的工具。当多个工具在功能上有重叠，或用途描述含糊时，agent 很容易搞不清该用哪一个。

命名空间，也就是给相关工具加上共同前缀，是一种帮助大量工具划分边界的方式。有些 MCP 客户端默认就会这么做。例如，按服务做命名空间，如 `asana_search`、`jira_search`，或按资源做命名空间，如 `asana_projects_search`、`asana_users_search`，都可以帮助 agent 在合适的时候选择合适工具。

我们发现，使用前缀命名空间还是后缀命名空间，会对工具使用评测产生并不小的影响。不同 LLM 的表现也可能不同，因此我们建议你通过自己的评测来决定采用哪种命名方案。

Agent 可能调用错误工具、用错误参数调用正确工具、调用工具数量不足，或者错误处理工具响应。通过有选择地实现那些名称能自然反映任务拆分方式的工具，你既能减少需要装入 agent 上下文中的工具和工具描述数量，也能把一部分原本需要在上下文里完成的 agentic 计算转移到工具调用本身，从而整体降低出错风险。

### 从工具中返回有意义的上下文

同样地，工具实现也应该注意：只把高信号信息返回给 agent。它们应优先考虑上下文相关性，而不是一味追求灵活性，并尽量避免返回底层技术标识符，例如 `uuid`、`256px_image_url`、`mime_type`。像 `name`、`image_url`、`file_type` 这样的字段，更有可能直接影响 agent 的后续行动与响应。

相比那些晦涩的标识符，agent 往往更擅长处理自然语言名称、术语或其他可解释标识。我们的经验是，仅仅把随意的字母数字 UUID 解析成更有语义、更易理解的名称，甚至只是一个从 0 开始的编号体系，都能显著提升 Claude 在检索任务中的准确率，并减少幻觉。

当然，在某些场景下，agent 也确实需要同时获取自然语言信息和技术标识，以便触发后续工具调用。例如，`search_user(name='jane')` 之后，可能还需要 `send_message(id=12345)`。对此，你可以在工具中加入一个简单的 `response_format` 枚举参数，让 agent 自己选择工具返回 `concise` 还是 `detailed` 响应。

你还可以提供更多格式选项，进一步增加灵活性，这有点像 GraphQL，允许你选择具体想收到哪些信息。下面是一个控制工具响应详细程度的 `ResponseFormat` 枚举示例：

```ts
enum ResponseFormat {
  DETAILED = "detailed",
  CONCISE = "concise"
}
```

一个详细工具响应的示例长度是 206 tokens，一个简洁工具响应的示例长度是 72 tokens。

在 Slack 中，线程和线程回复由唯一的 `thread_ts` 标识，而获取回复时必须用到它。像 `thread_ts` 以及其他 ID，例如 `channel_id`、`user_id`，都可以从 `detailed` 工具响应中拿到，以支持后续必须依赖这些 ID 的工具调用。而 `concise` 工具响应则只返回线程内容，不包含这些 ID。在这个例子中，使用 `concise` 响应只消耗了大约三分之一的 tokens。

甚至工具响应的结构形式，例如 XML、JSON 或 Markdown，也可能影响评测表现，并不存在放之四海而皆准的统一答案。这是因为 LLM 基于下一个 token 预测进行训练，往往会对更贴近其训练分布的格式表现更好。最优的响应结构会随任务和 agent 的不同而有很大差异，因此我们建议你仍然通过自己的评测来选择最合适的结构。

### 为 token 效率优化工具响应

优化上下文的质量很重要，而优化工具返回给 agent 的上下文数量同样重要。

对于那些可能消耗大量上下文的工具响应，我们建议默认实现分页、范围选择、过滤和/或截断等机制，并设置合理的默认参数。以 Claude Code 为例，我们默认将工具响应限制在 25,000 tokens 以内。我们预计，未来 agent 的有效上下文长度会继续增长，但对上下文高效率工具的需求不会消失。

如果你选择对响应做截断，一定要提供有帮助的引导信息。你可以直接鼓励 agent 采用更省 token 的策略，例如在知识检索任务中，执行多次小范围、定向搜索，而不是做一次宽泛的大搜索。同样地，如果某次工具调用报错，比如输入验证失败，那么错误响应也可以通过 prompt engineering 进行设计，使其清楚传达具体且可执行的改进建议，而不是仅仅返回一个晦涩错误码或 traceback。

截断响应和错误响应本身，都可以把 agent 引导到更节省 token 的工具使用方式上，例如使用过滤条件或分页，也可以顺便给出正确格式输入的示例。

### 对工具描述做 prompt engineering

现在来到提升工具表现最有效的方法之一：对工具描述和规格做 prompt engineering。由于这些内容会直接加载进 agent 的上下文，它们会共同影响 agent 的工具调用行为。

编写工具描述和规格时，可以想象你是在向团队里一位刚入职的新同事介绍这个工具。想一想你自己默认知道、但对新同事并不显然的内容，例如专门的查询格式、小众术语的定义、底层资源之间的关系，并把这些隐含背景显式写出来。要通过清晰描述输入和输出，并用严格数据模型去约束，来避免歧义。

尤其是，输入参数命名必须明确无歧义。比如，与其使用 `user` 这样含糊的参数名，不如使用 `user_id`。

借助评测，你就能更有把握地衡量这些 prompt engineering 是否真的带来提升。哪怕只是对工具描述做很小的改动，也可能产生非常显著的效果。我们在对工具描述做出精细调整后，Claude Sonnet 3.5 在 `SWE-bench Verified` 评测上达到了当时最先进的表现，错误率明显下降，任务完成效果显著提升。

你还可以在开发者指南中找到更多关于工具定义的最佳实践。如果你是在为 Claude 构建工具，我们也建议了解工具是如何被动态加载进 Claude 的 system prompt 中的。最后，如果你是在为 MCP server 编写工具，那么 tool annotations 可以帮助说明哪些工具需要开放世界访问权限，或者会执行破坏性变更。

## 展望未来

为了给 agent 构建高效工具，我们需要把软件开发实践从“面向可预测、确定性模式”重新调整为“面向非确定性模式”。

通过本文描述的这种迭代式、评测驱动式流程，我们总结出了一些稳定模式：高效工具通常有意图明确、定义清晰、节制使用 agent 上下文、能够被灵活组合进不同工作流，并能让 agent 以更直观的方式解决真实任务。

未来，agent 与世界交互的具体机制还会继续演化，无论是 MCP 协议的更新，还是底层 LLM 本身的升级。但只要坚持以系统化、评测驱动的方法去持续改进 agent 的工具，我们就能确保，随着 agent 能力提升，它们使用的工具也会同步进化。
