# Introducing advanced tool use on the Claude Developer Platform - 笔记

发布时间：`2025-11-24`

## 文章主旨

这篇文章主要介绍了 Claude Developer Platform 新增的三项高级工具使用能力：`Tool Search Tool`、`Programmatic Tool Calling` 和 `Tool Use Examples`。它想解决的核心问题是，随着 agent 接入的工具越来越多、工作流越来越长、参数结构越来越复杂，传统的“把全部工具定义提前塞进上下文，再靠自然语言一轮轮调用工具”的方式已经越来越低效。Anthropic 给出的方向是：让 agent 按需发现工具、用代码而不是纯自然语言来编排工具调用，并通过示例而不是只靠 schema 来学习正确用法。

## 核心观点

### 1. 大规模工具系统的瓶颈，首先是上下文，而不是能力本身

文章一开始就指出，当 agent 面对数十个 MCP server 和数百个工具时，最大的问题并不只是“有没有这些工具”，而是“这些工具定义本身已经把上下文吃掉了”。如果在用户请求还没开始前，工具定义就先消耗掉几十万 tokens，那么模型就算理论上拥有完整能力，实际上也会因为上下文空间被挤占而表现下降。这个观点很重要，因为它把工具系统的优化重点从“继续加能力”转到了“如何让能力以更轻量的形式被访问”。

### 2. Tool Search Tool 的本质，是把“全部加载”改成“按需发现”

`Tool Search Tool` 解决的问题不是工具不够多，而是工具太多时，不能再把所有定义一股脑塞给模型。它的核心思想是让 Claude 先只加载搜索工具和极少数高频工具，其余工具通过 `defer_loading` 延迟加载，等需要时再搜索并展开。这个设计的价值在于，它让 Claude 保留对完整工具库的访问权，同时只为当前任务真正相关的那几个工具付上下文成本，本质上是一种把“工具访问”从静态注入改成动态检索的机制。

### 3. Tool Search Tool 不只是省 token，也能提高选工具准确率

文章特别强调，工具过多不仅意味着上下文膨胀，还会提高选错工具和传错参数的概率，尤其是当工具名很相似时。Tool Search Tool 通过缩小当前可见工具集合，减少了 Claude 面前的“候选噪音”，从而在大工具库场景下提升了准确率。这个观点很有价值，因为它说明“少看到一些无关工具”不只是效率优化，也是在降低决策复杂度，本质上是在改善 agent 的检索和选择环境。

### 4. Programmatic Tool Calling 的核心，不是“能在代码里调工具”，而是把编排逻辑移出模型上下文

`Programmatic Tool Calling` 看上去像是让 Claude 通过 Python 调工具，但文章真正想表达的是：复杂工作流里的大量中间结果和编排逻辑，不应该都挤在模型上下文里完成。通过让 Claude 写代码来调用多个工具、处理结果、做循环和条件判断，很多原本会不断污染上下文的中间步骤都被放进了代码执行环境里。这个观点非常关键，因为它意味着 agent 的“思考空间”和“执行空间”被拆开了，模型只需要关注最终有意义的结果，而不必背负所有过程数据。

### 5. 中间结果不进入上下文，是 PTC 最大的价值之一

文章用预算检查的例子说明了这一点：如果团队成员、费用明细和预算限制全都进上下文，Claude 需要自己在自然语言层面逐步求和、比对和归纳；而用 `Programmatic Tool Calling` 时，这些步骤都在代码里完成，Claude 最终只看到“哪几个人超预算了”这一小份结果。这个观点的本质是，很多任务的难点并不在最终判断，而在前面的数据搬运和整理；只要把这些环节从上下文里拿出去，模型就能把更多注意力留给真正需要判断的部分。

### 6. Programmatic Tool Calling 也在减少推理轮次，而不只是减少 token

文章指出，传统工具调用每一步都要重新触发一次模型推理，这会导致大量延迟和额外误差。而当 Claude 先写出一段完整的编排代码时，后续多个工具调用就可以由代码执行环境连续完成，不需要每次都回到模型层做新的推理。这个观点说明，PTC 的收益不仅是“更省上下文”，还包括“更少模型往返”和“更明确的控制流”，这对于多步、多对象、可并行的任务尤其重要。

### 7. Tool Use Examples 解决的是 schema 解决不了的“行为歧义”

文章对 `Tool Use Examples` 的定位非常清楚：JSON Schema 可以告诉 Claude 什么字段合法、哪些字段必填，但没法告诉它什么时候该填哪些可选字段、某些参数组合在业务上意味着什么、日期和 ID 应该遵循什么格式。也就是说，schema 解决的是“结构合法性”，而例子解决的是“实际使用习惯”。这个观点很有启发性，因为它说明在复杂工具场景里，正确调用往往不是靠约束规则 alone，而是靠示范调用模式来学习。

### 8. 示例的价值，不在于多，而在于能体现模式

文章给出的工单 API 例子展示了一个重要思想：少量高质量示例就足以让 Claude 学到日期格式、ID 风格、嵌套对象出现条件，以及不同优先级与升级参数之间的关系。也就是说，示例最重要的功能不是覆盖所有输入，而是让模型看懂“什么情况该怎么填”。这个观点很实用，因为它提醒我们写例子时不必追求大全，而应优先展示那些 schema 无法清楚表达、却会显著影响调用正确性的模式。

### 9. 这三项能力分别解决不同瓶颈，应当按问题分层引入

文章在最佳实践部分给出一个很实用的判断框架：如果你的主要问题是工具定义太多导致上下文膨胀，就优先上 `Tool Search Tool`；如果问题是中间结果太大、流程太长，就优先上 `Programmatic Tool Calling`；如果问题是参数老传错、调用格式总出错，就优先上 `Tool Use Examples`。这个观点的价值在于，它把这三项能力从“都很高级的新特性”重新拆回成“分别处理不同瓶颈的工程部件”，有助于避免一上来就全量引入复杂性。

### 10. 三项能力合起来，分别对应“找对工具、用高效方式执行、正确传参数”

文章最后隐含的一条主线是，这三项能力其实正好覆盖了工具使用链路上的三个关键环节。`Tool Search Tool` 负责让 Claude 找到对的工具，`Programmatic Tool Calling` 负责让工具执行过程更高效，`Tool Use Examples` 则负责让 Claude 在调用时更少出错。这个观点很值得记住，因为它把看似分散的三个 beta 功能整合成了一套完整的工具使用升级路径。

## 简要结论

如果把全文压缩成一句话，它真正想表达的是：`当 agent 开始操作大规模工具库和复杂工作流时，工具使用必须从“静态加载 + 自然语言逐步调用”升级成“动态发现 + 代码编排 + 示例驱动调用”。` 这篇文章的价值不只是介绍了三个新能力，更重要的是说明了未来 agent 工具系统的设计方向：减少上下文浪费、把编排逻辑移出模型、并让正确调用方式可以被显式学习。
