# 深入 Claude Code：泄露源码究竟揭示了什么 - 笔记

## 文章主旨

这篇文章基于作者对一次 Claude Code 调试 source map 泄露事件的描述，整理了 Claude Code 的内部架构：它以 `QueryEngine.ts`、`Tool.ts` 和 `commands.ts` 为核心，围绕模型编排、工具执行、命令系统、记忆管理、权限控制、feature flag 和多 Agent 能力展开。作者的核心判断是，Claude Code 并不只是一个终端聊天式编码助手，而是在朝一个长期在线、主动工作、能并行调度子 Agent 的持久型开发伙伴演化。

## 核心观点

### Claude Code 的架构核心集中在三个文件

原文称，`QueryEngine.ts` 负责每次模型交互的编排，`Tool.ts` 定义 40 多种 Agent 能力及其权限 schema，`commands.ts` 实现斜杠命令系统。这三个文件构成 Claude Code 的主要结构支柱：一个决定模型如何思考和循环，一个决定它能调用什么能力，一个决定用户如何通过终端命令控制它。

### QueryEngine 的重点是循环、流式解析和上下文管理

作者把 `QueryEngine.ts` 描述为 Claude Code 的心脏：它在 API 调用前组装记忆、工具 schema、项目上下文和对话历史；在响应流式返回时边接收边解析；遇到工具调用就排队执行，再把结果追加回上下文继续循环。它还会在上下文触顶前做压缩，而不是简单截断旧消息，这让长时间任务中的早期决策仍能以摘要形式保留下来。

### 工具系统以结构化 schema 和权限分层降低风险

文章认为 Claude Code 的工具能力不是随意执行文本命令，而是由工具名、模型可读描述、JSON Schema 和执行函数组成。模型输出结构化 JSON 后，引擎先验证再执行。权限按风险划分：读文件可自动批准，写文件和 bash 命令需要确认，受限环境中某些工具会被完全阻止。即便项目目录被标记为可信，shell 命令仍要经过注入和路径遍历检查。

### 记忆架构的关键是索引、主题文件和转录分离

原文称 Claude Code 不会把所有历史都塞进上下文，而是每次会话只加载 `CLAUDE.md` 和 `MEMORY.md`。其中 `MEMORY.md` 更像索引，保存短指针；真正内容放在按主题拆分的文件中，只在相关时加载；完整会话转录不直接进入上下文，只在需要时通过 grep 检索。这个设计让持久记忆不会随着会话数量线性吞噬 token。

### AutoDream 用后台子 Agent 做记忆整合

作者提到，五次以上会话后，`AutoDream` 会每 24 小时整合一次记忆：批处理历史、合并重复项、删除矛盾、把模糊笔记变成具体事实，并把索引裁剪到 200 行以内。它运行在工具受限的 fork 子 Agent 中，与主会话隔离，目标是在不污染活跃上下文的情况下维护长期记忆质量。

### 记忆系统主动避免存储容易过时的派生事实

文章强调一个重要设计原则：Claude Code 不把代码结构、文件路径、PR 历史等容易过时的派生事实写入长期记忆，因为这些内容可以从当前源码重新推导。记忆被当作线索而非事实，模型被要求在使用前根据当前项目状态验证。这是持久型 Agent 避免“记住错误过去”的关键约束。

### Feature flag 暗示产品正在走向自主和多 Agent

作者认为，代码中的 feature flag 不是小实验，而是隐藏完整子系统的发布开关。`KAIROS` 指向始终在线的后台自主 Agent，`COORDINATOR_MODE` 指向并行 worker agent 的编排，`PROACTIVE` 则介于普通响应式模式和完整 daemon 之间，通过周期性 tick prompt 让 Claude 在消息之间主动工作。这些 flag 共同暗示 Claude Code 的方向是从“你问我答”转向“持续观察并主动推进工作”。

### 内部模型代号和质量注释暴露了产品化约束

原文称源码中出现了 `Capybara`、`Fennec`、`Numbat`、`opus-4-7`、`sonnet-4-8` 等未公开代号或版本字符串，并包含关于虚假声明率、过度注释、提示调优等内部质量备注。作者借此说明，真实产品中的模型使用不只是选择一个模型名，而是围绕缺陷、倾向和风险做提示约束、工具限制和体验设计。

### BUDDY 系统体现“工具变伙伴”的产品方向

文章把 BUDDY 描述成一个带稀有度、属性、装饰和变体的 companion 系统。作者认为这不是孤立的游戏化设计，而是与 KAIROS、AutoDream、多 Agent 编排等方向一致：如果 Claude Code 会长期在后台陪伴开发者工作，那么产品就可能从工具转向更具持续陪伴感的工作伙伴。

### 最终方向是持久、主动、可并行的开发 Agent

作者最后的判断是，Claude Code 的源码显示它正在从终端编码助手演变成一种持续运行的开发 Agent：它会维护长期记忆，监听仓库事件，主动寻找可做的工作，协调多个子 Agent，并通过权限系统管理风险。文章的价值不在于复述泄露事件本身，而在于用这些架构线索推断 Claude Code 的产品演化方向。
