# Effective context engineering for AI agents - 中文翻译

发布时间：`2025-09-29`

上下文是 AI agent 的关键资源，但它也是有限的。在这篇文章中，我们会探讨如何有效地筛选和管理驱动 agent 的上下文。

在应用型 AI 领域，经过几年大家都把注意力放在 prompt engineering 上之后，一个新词开始受到更多关注：`context engineering`。如今，基于语言模型进行构建，重点已经不再只是为 prompt 找到恰当的词句，而是转向一个更广泛的问题：`怎样配置上下文，最有可能让模型产生我们期望的行为？`

上下文，指的是在对大语言模型（`LLM`）进行采样时，一同输入进去的那组 tokens。这里的工程问题，是如何在 LLM 固有约束之下，优化这些 tokens 的效用，从而稳定地得到理想结果。要真正驾驭 LLM，往往要求我们“在上下文中思考”，也就是从整体上考虑：某一时刻 LLM 能看到的完整状态是什么，以及这个状态可能诱发出哪些行为。

在这篇文章中，我们将探讨这门新兴的“上下文工程”艺术，并给出一种更成熟的心智模型，用来构建可控且高效的 agent。

## Context engineering 与 prompt engineering

在 Anthropic，我们把 `context engineering` 看作 `prompt engineering` 的自然延伸。`Prompt engineering` 指的是：为了获得最佳结果，如何编写和组织 LLM 指令。`Context engineering` 指的是：在 LLM 推理过程中，如何筛选和维护一组最优 tokens，也就是最优信息集合，这其中不仅包括 prompt 本身，也包括所有其他可能进入上下文的信息。

在 LLM 工程的早期阶段，prompt 是 AI 工程工作的核心部分，因为那时绝大多数用例仍停留在单轮分类或单轮文本生成等任务上。正如这个词本身所暗示的，prompt engineering 的重点在于如何写出有效 prompt，尤其是 system prompt。但随着我们开始构建更强大的 agent，让它们在多轮推理和更长时间跨度上持续工作，我们就必须开始管理整个上下文状态，包括 system instructions、tools、`Model Context Protocol`（`MCP`）、外部数据、消息历史等。

一个在循环中运行的 agent 会不断产生越来越多、可能对下一轮推理有帮助的数据，而这些信息必须被反复筛选和提炼。`Context engineering` 就是在这个不断变化、可能进入上下文的信息宇宙中，挑选出真正该放进有限上下文窗口里的内容的艺术与科学。

与“写一个 prompt”这种离散任务不同，`context engineering` 是迭代性的：每一次我们决定要把哪些信息传给模型时，都会再次发生“筛选上下文”这一过程。

## 为什么 context engineering 对构建强大的 agent 很重要

尽管 LLM 越来越快，也能处理越来越长的数据，我们依然观察到，它们和人类一样，到了一定程度就会失焦或混乱。很多类似“大海捞针”的基准测试都揭示了一个现象：`context rot`。也就是说，随着上下文窗口中的 token 数量增加，模型从这些上下文中准确回忆信息的能力会下降。

有些模型的退化更平缓，有些更明显，但这种特征在所有模型中都会出现。因此，上下文必须被视为一种有限资源，而且其边际收益会递减。就像人类有有限的工作记忆容量一样，LLM 也有一个“注意力预算”，它在解析大量上下文时会不断消耗这笔预算。每引入一个新 token，都会消耗其中一部分，这也让我们更需要认真筛选真正该提供给模型的 tokens。

这种注意力稀缺，根源在于 LLM 的架构限制。LLM 基于 Transformer 架构，允许每个 token 与上下文中的其他所有 token 相互注意。对于 `n` 个 tokens，这意味着会产生 `n²` 个两两关系。

随着上下文长度增加，模型对这些两两关系的建模能力会被不断拉薄，于是在“上下文更长”和“注意力更集中”之间形成天然张力。此外，模型的注意力模式来自训练数据分布，而训练数据中通常短序列远比长序列常见。这意味着模型对超长上下文的经验更少，也缺乏足够多专门为跨上下文依赖设计的参数。

像位置编码插值这样的技术，虽然能帮助模型适配更长序列，把它们映射回原本训练时更小的上下文，但这也会带来一定程度的 token 位置理解退化。这些因素共同造成的不是一个陡峭的性能悬崖，而是一条性能梯度：模型在长上下文下仍然能工作，但在信息检索和长距离推理上，精度可能会弱于短上下文时的表现。

正因为如此，要构建强大的 agent，认真做 `context engineering` 就变得非常必要。

## 有效上下文的结构

既然 LLM 受限于有限的注意力预算，那么好的 `context engineering` 就意味着：找到尽可能小、但信号最强的一组 tokens，让它们最大化产生目标行为的概率。说起来简单，做起来很难。下面我们会分不同上下文组件，说明这个原则在实践中意味着什么。

`System prompt` 应该极其清晰，语言要简单直接，并以恰当的“抽象高度”表达信息。所谓合适的高度，就是避免落入两个常见失败模式中的任意一端。一端是工程师在 prompt 中硬编码非常复杂、脆弱的逻辑，试图精确规定 agent 的每一个行为，这会让系统极不稳健，也让维护成本越来越高。另一端则是给出过于模糊、过于高层的指导，既没提供足够具体的行为信号，也错误地假设模型与你共享了某些背景。理想的 system prompt 应该在两者之间取得平衡：具体到足以有效引导行为，但又灵活到能给模型提供强有力的启发式，而不是脆弱的 if-else 规则。

我们建议把 prompt 组织成若干清晰的部分，比如 `<background_information>`、`<instructions>`、`## Tool guidance`、`## Output description` 等，并使用 XML 标签或 Markdown 标题去划分这些部分。虽然随着模型能力提高，具体格式的重要性可能会下降，但在当前阶段，这仍然是有帮助的。

无论你如何组织 system prompt，都应努力追求“最小但完整的信息集合”，让它足以定义你期望的行为。这里的“最小”并不一定意味着“最短”，因为你仍然必须在前面给 agent 足够的信息，确保它能遵守目标行为。最好的做法是：先用当前最强模型测试一个尽可能简洁的 prompt，看看任务效果如何；然后根据初始测试中暴露出来的失败模式，再逐步添加更清晰的说明和示例。

`Tools` 让 agent 能与环境交互，并在工作过程中拉入新的上下文。因为工具定义了 agent 与其信息/行动空间之间的契约，所以工具必须促进效率：既要返回 token 上高效的信息，也要鼓励 agent 采用高效的行为路径。

在《Writing tools for AI agents – with AI agents》中，我们讨论过如何构建既容易被 LLM 理解、又尽量避免功能重叠的工具。和一个良好代码库中的函数一样，工具应当是自包含的、对错误稳健的，并且在用途上极其清晰。输入参数也应该具有描述性、无歧义，并且顺应模型本身的天然优势。

我们最常见到的失败模式之一，就是工具集合臃肿，覆盖了太多功能，或者在“该用哪个工具”这件事上引发模糊决策点。如果一个人类工程师都无法明确判断某个场景下到底该选哪个工具，那么我们也不应期待 AI agent 表现得更好。正如后文会讲到的，给 agent 提供一组“最小可用”的工具，不仅能提高可靠性，也有利于在长对话过程中维护和修剪上下文。

提供示例，也就是 `few-shot prompting`，依然是我们强烈建议的最佳实践之一。不过，很多团队会把大量边缘案例一股脑塞进 prompt，试图用它来表达 LLM 在该任务上应遵守的所有规则。我们不建议这样做。相反，我们建议精心筛选出一组多样化且具有代表性的标准示例，来有效展示 agent 应有的行为。对 LLM 来说，示例就像“胜过千言万语的图片”。

因此，我们对于上下文各个组成部分——system prompts、tools、examples、message history 等——的总体建议是：要深思熟虑，让上下文既有信息密度，又保持紧凑。接下来我们将讨论运行时如何动态检索上下文。

## 上下文检索与 agentic search

在《Building effective AI agents》中，我们强调过基于 LLM 的 workflow 与真正的 agent 之间的区别。自那篇文章之后，我们越来越倾向于使用一个更简洁的定义：`agent = 在循环中自主使用工具的 LLM`。

在与客户合作的过程中，我们看到整个领域正逐渐收敛到这一简单范式上。随着底层模型越来越强，agent 的自主性也得以提升：更聪明的模型允许 agent 独立地在复杂问题空间中导航，并从错误中恢复。

我们也开始看到，工程师设计 agent 上下文的方式正在发生变化。如今，很多原生 AI 应用都会采用某种基于 embedding 的、发生在推理之前的检索机制，把重要上下文预先取出来供 agent 推理。而随着系统变得更 agentic，我们越来越多地看到团队在这些机制之上，再叠加“`just in time`”的上下文策略。

所谓 “just in time” 策略，就是不再把所有相关数据都提前处理好，而是让 agent 持有一些轻量级引用，例如文件路径、存储的查询、网页链接等，再通过工具在运行时按需把真实数据加载进上下文。Anthropic 的 agentic 编程解决方案 Claude Code 就采用了这种方式，用来对大型数据库执行复杂数据分析。模型可以编写定向查询、存储结果，并利用 `head`、`tail` 等 Bash 命令分析大体量数据，而无需把完整数据对象都加载进上下文。这种方式很像人类的认知方式：我们通常不会把整套知识体系死记硬背，而是借助文件系统、收件箱、书签等外部组织与索引机制，在需要时再提取相关信息。

除了存储效率之外，这些引用自身的元数据也提供了一种高效修正行为的机制，不管这种机制是显式的还是直觉式的。对于一个在文件系统中工作的 agent 来说，`tests` 文件夹里的 `test_utils.py` 和 `src/core_logic/` 里的同名文件，显然意味着完全不同的用途。目录层级、命名规范和时间戳，都能提供重要信号，帮助人类和 agent 判断何时以及如何使用某条信息。

让 agent 自主地浏览并检索数据，还能带来“`progressive disclosure`”，也就是渐进式披露上下文的能力。换句话说，agent 可以通过探索，一层层地逐步发现相关上下文。每一次交互都会产生能影响下一步判断的新上下文：文件大小暗示复杂度，命名方式暗示用途，时间戳则可能是相关性的代理信号。这样，agent 就能一层层地构建理解，只在工作记忆里维持真正必要的内容，同时把其他信息交给记笔记等外部持久化策略。这种自我管理上下文窗口的方式，能让 agent 始终聚焦于相关子集，而不至于被详尽但可能无关的信息淹没。

当然，这里存在权衡：运行时探索的速度，通常比直接检索预先计算好的数据更慢。不仅如此，还需要有明确而周到的工程设计，确保 LLM 拥有合适的工具和启发式，能高效地在其信息环境中导航。如果缺乏正确引导，agent 可能会因为误用工具、走入死胡同或错过关键信息，而浪费大量上下文。

在某些场景里，最有效的 agent 可能会采用一种混合策略：先为了速度预取部分数据，再在必要时自主做进一步探索。“自主程度该设置到什么水平”，要取决于任务本身。Claude Code 就采用了这种混合模式：它会把 `CLAUDE.md` 文件直接、朴素地放进上下文，同时又利用 `glob` 和 `grep` 等原语，在需要时即时浏览环境并检索文件，从而有效绕开静态索引过时和复杂语法树维护的问题。

这种混合策略也许更适合那些内容变化没那么频繁的场景，比如法律和金融。随着模型能力不断提升，agent 设计会越来越倾向于“让聪明模型以更聪明的方式行动”，并逐渐减少人为筛选。在这个变化飞快的领域里，对构建 Claude agent 的团队来说，我们最常给出的建议仍然是：`做那个最简单但能奏效的方案。`

## 面向长时任务的 context engineering

长时任务要求 agent 在一连串操作中维持连贯性、上下文和目标导向行为，而整个过程的 token 数量往往会超过 LLM 的上下文窗口。对于那些持续几十分钟到数小时的任务，比如大规模代码库迁移或全面研究项目，agent 需要专门技术来绕开上下文窗口大小的限制。

一种直觉性的做法，似乎是等更大的上下文窗口出现。但在可预见的未来，哪怕上下文窗口越来越大，它仍然会面临上下文污染和信息相关性问题，至少在那些追求最强 agent 表现的场景中是如此。为了让 agent 在长时间跨度上高效工作，我们开发了几种直接应对这些问题的技术：`compaction`、`structured note-taking` 和 `multi-agent architectures`。

### Compaction

`Compaction` 指的是：当一段对话即将触达上下文窗口上限时，先对其内容做总结，再以这段总结为基础重新开启一个新的上下文窗口。通常来说，compaction 是 context engineering 中第一道用于改善长期连贯性的杠杆。它的核心，是以尽可能高保真的方式提炼当前窗口内容，使 agent 可以在性能几乎不明显下降的前提下继续工作。

例如在 Claude Code 中，我们会把消息历史传给模型，让它总结并压缩最关键的信息。模型会保留架构决策、尚未解决的 bug 和实现细节，同时丢弃冗余的工具输出或重复消息。随后，agent 会在这份压缩上下文基础上继续工作，同时保留最近访问过的五个文件。这样一来，用户就能获得连续体验，而无需关心上下文窗口本身的限制。

Compaction 的艺术，在于选择保留什么、丢弃什么。如果压缩得过于激进，就可能丢掉那些表面上不起眼、但在后面才暴露出重要性的关键信息。对于实现 compaction 系统的工程师，我们建议在复杂 agent trace 上仔细调 prompt。起步时先最大化召回，确保 compaction prompt 能抓到所有相关信息；之后再逐步提高精度，把多余内容剔除掉。

一个很容易处理的冗余内容，就是清空深层历史中的工具调用及其结果：当某次工具调用已经远在消息历史深处时，agent 为什么还需要再次看到原始结果？因此，最安全、最轻量的一种 compaction 形式之一，就是清除旧工具结果，这也已经作为 Claude Developer Platform 的新功能对外提供。

### Structured note-taking

`Structured note-taking`，或者说 `agentic memory`，是一种让 agent 定期把笔记写入上下文窗口之外的持久存储中的技术，这些笔记之后还会在需要时重新拉回上下文。

这种策略以极低的开销提供了持久记忆。比如 Claude Code 会创建待办清单，或者你的自定义 agent 维护一个 `NOTES.md` 文件。这样简单的模式，却允许 agent 在复杂任务中跟踪进度，维持那些否则会在几十次工具调用之后丢失的关键上下文和依赖关系。

`Claude playing Pokémon` 很好地展示了记忆是如何改变 agent 在非编程领域的能力的。这个 agent 能在上千步游戏过程中精确记账，例如记录“过去 1,234 步里，我一直在 Route 1 训练我的 Pokémon，Pikachu 已经从目标 10 级中的 8 级提升上来了”。即使没有人提示它该如何组织记忆，它也会自发地绘制探索过的区域地图，记住已经解锁了哪些关键成就，并保留关于战斗策略的笔记，帮助自己学习哪些攻击对不同对手最有效。

在上下文重置之后，agent 会读回自己写的笔记，然后继续数小时的训练或地牢探索。正是这种跨总结步骤的连贯性，让长时策略成为可能，而如果一切都只保留在 LLM 当前上下文中，这几乎无法做到。

作为 Sonnet 4.5 发布的一部分，我们在 Claude Developer Platform 上公开 beta 推出了 memory tool，让 agent 更容易通过文件系统在上下文外存储和读取信息。这使 agent 可以随着时间积累知识库、跨会话维持项目状态，并在无需把一切都保留在上下文中的前提下引用过去的工作。

### Sub-agent architectures

`Sub-agent architectures` 是另一种绕开上下文限制的方法。与其让一个 agent 独自维持整个项目的状态，不如让多个专门的 sub-agent 在干净的上下文窗口中处理聚焦任务。主 agent 负责高层规划，而 sub-agent 则执行深入的技术工作，或使用工具去寻找相关信息。每个 sub-agent 都可以在自己的上下文中做大量探索，消耗成千上万 tokens 甚至更多，但最终只返回一份压缩且提炼过的总结，通常是 1,000 到 2,000 tokens。

这种方式实现了清晰的职责分离：详细的检索上下文被隔离在 sub-agent 内部，而主 agent 则专注于综合和分析结果。这个模式在《How we built our multi-agent research system》中有详细讨论，并且在复杂研究任务上，相比单 agent 系统带来了显著提升。

该如何在这些方法之间做选择，要取决于任务特征。例如：

- `Compaction` 更适合需要大量往返对话的任务，因为它有利于维持会话流；
- `Note-taking` 更适合具有清晰里程碑的迭代开发；
- `Multi-agent architectures` 更适合复杂研究和分析任务，因为并行探索会带来明显收益。

即使模型能力持续提升，如何在长时间交互中维持连贯性，仍将是构建更强大 agent 的核心挑战。

## 结论

`Context engineering` 代表了我们构建 LLM 应用方式上的一次根本性转变。随着模型能力越来越强，挑战不再只是写出完美 prompt，而是在每一步中认真筛选，决定哪些信息该进入模型有限的注意力预算。无论你是在为长时任务实现 compaction、设计 token 高效的工具，还是让 agent 在运行时按需探索环境，核心原则始终不变：找到那组尽可能小、但信号最强的 tokens，让它们最大化地产生你想要的结果。

随着模型进步，这些技术还会继续演化。我们已经看到，更聪明的模型往往需要更少的刚性工程控制，让 agent 能以更高自主性运作。但即便能力继续扩展，把上下文视作一种珍贵且有限的资源，仍将是构建可靠、高效 agent 的中心原则。
