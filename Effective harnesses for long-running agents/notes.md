# Effective harnesses for long-running agents - 笔记

发布时间：`2025-11-26`

## 文章主旨

这篇文章主要讲的是，长时间运行的 agent 并不是简单把模型放进循环里就能稳定工作，因为每跨过一个上下文窗口，新的 agent session 本质上都会像“失忆接班”一样重新开始。Anthropic 发现，仅靠 compaction 还不足以保证长期稳定推进，于是提出了一种更实用的 harness 设计：先用一个 `initializer agent` 在项目最开始搭好工作环境和进度记录机制，再让之后的 `coding agent` 每次只做增量进展，并在结束时把环境收拾到适合下一轮继续接手的干净状态。

## 核心观点

### 1. 长时间运行 agent 的核心难点，不是推理本身，而是跨 session 的“接班问题”

文章最重要的判断是，很多长时任务失败，并不是因为单轮上下文里的模型不会做，而是因为任务必须拆到多个离散 session 里完成，而每个新 session 都没有天然记忆。这个问题很像一个项目由轮班工程师接手，但每个接班人都不知道上一班做了什么。这个观点很关键，因为它把 long-running agent 的本质问题重新定义成了“如何让工作状态可被持续传承”，而不只是“如何让单次推理更强”。

### 2. 单靠 compaction 不能保证长期工程任务稳定推进

文章明确指出，虽然 Claude Agent SDK 已经有 compaction 这类上下文管理能力，但在真实复杂项目中，仅靠它并不足以让 agent 跨多个上下文窗口稳定构建生产级应用。因为 compaction 即使保留了摘要，也未必能把项目状态、未完成功能、已知 bug 和后续意图传达得足够清楚。这个观点很重要，因为它说明对长时工程任务来说，“压缩上下文”只是基础设施，还不是完整解决方案。

### 3. 失败模式一：agent 容易一次做太多，结果把项目留在半成品状态

Anthropic 观察到的第一个典型失败是，agent 会倾向于一口气做完整个应用，而不是一步步推进。这样一来，它很容易在上下文耗尽时停在一个半实现、半损坏、又没有文档说明的状态里，导致下一轮 agent 既不知道它做到哪，也不知道哪些代码可信。这个观点很有价值，因为它揭示了一个长时 agent 的常见风险：不受约束的雄心会直接破坏可接续性。

### 4. 失败模式二：agent 在项目做到一半时，容易误以为“已经完成”

另一个典型失败是，后续 agent 进入项目后，看到已经有不少代码和功能存在，就误判为任务已经完成。这个问题的根源并不是模型懒惰，而是项目“完成”的定义没有被结构化地显式表达出来。这个观点非常重要，因为它说明在长时任务里，必须把“最终还有哪些事没做完”外部化成一个可读取、可更新、可验证的状态，而不能只靠 agent 自己临场判断。

### 5. 一个有效 harness 需要把“初始化环境”和“持续增量开发”拆成两个角色

Anthropic 最终采用的是双组件方案：第一个 `initializer agent` 负责搭建初始环境，包括运行脚本、进度文件、初始 git 历史和功能列表；之后每个 `coding agent` 都只负责在既有环境里做小步推进，并把状态整理好留给下一轮。这个观点的价值在于，它说明长时任务的关键不是让所有 agent 做同一种事，而是让第一个 agent 先把项目的可维护性和可传承性基础搭好。

### 6. 功能列表的真正作用，不只是列需求，而是外部化“什么才算完成”

文章提到，他们让 initializer agent 先写出一个覆盖完整功能需求的 JSON 文件，把每个功能都先标记为 `passes: false`，后续 agent 只能通过测试后把这个字段改成 `true`。这个设计的意义非常大，因为它把“项目是否完成”的判断从主观感受，变成了一个可以逐项核对的外部状态机。换句话说，feature list 不只是需求文档，更是长时 agent 的进度坐标系。

### 7. 强制增量推进，是抑制 agent 失控扩张的关键手段

文章明确说，真正让系统开始稳定的关键之一，是要求 coding agent 一次只做一个功能。这个约束听起来朴素，但作用非常大，因为它直接切断了 agent 想“一步到位”的倾向，也减少了每轮结束时留下大面积半成品的概率。这个观点很重要，因为它说明对 long-running agent 来说，速度感往往来自稳定的可累计进展，而不是单轮里试图完成最多工作。

### 8. Git commit 和 progress 文件，本质上是让 agent 学会像工程师一样交接班

文章发现，最有效的做法之一，是要求每个 coding agent 在 session 结束时写出清晰的 git commit 和进度总结。这样，下一轮 agent 不需要再猜测之前发生了什么，也能在必要时用 git 回滚坏修改。这个观点的价值在于，它表明长时 agent 的核心不是发明一种全新记忆机制，而是复用人类工程实践里已经非常成熟的“交接、提交、记录”制度。

### 9. 测试不只是验证当前改动，更是检查环境有没有被上一轮悄悄弄坏

文章在测试部分的一个关键洞见是，每轮 coding agent 开始时都应该先启动项目、做基础端到端验证，而不是直接开始写新功能。原因是，如果当前环境本身已经坏了，而 agent 继续叠加新改动，问题只会越来越复杂。这个观点非常实用，因为它把测试从“改完之后验收自己做的东西”，扩展成了“接班之前先确认工地本身是安全的”。

### 10. 端到端测试工具对于 long-running web agent 尤其关键

Anthropic 观察到，如果不明确要求，Claude 往往会满足于单元测试、curl 请求或局部验证，却没有意识到功能是否真的像用户一样从头到尾可用。一旦引入 Puppeteer 这样的浏览器自动化工具，并 prompt 它像真人那样测试，性能会明显提升。这个观点很重要，因为它说明长时 web agent 的很多失败并不来自代码层逻辑，而来自“以为功能可用、其实用户路径并没真正通”的错觉。

### 11. 一个好的 long-running harness，本质上是在降低新 session 的启动成本

文章描述的 session 开头流程很机械：`pwd`、看 `progress`、看 `feature_list`、看 `git log`、读 `init.sh`、起服务、跑基础测试。但正是这种固定流程，让每个新 session 都能快速进入可工作状态，而不用重新摸索目录、运行方式、最近改动和当前优先任务。这个观点的价值在于，它说明长时 agent 的效率不只是体现在“每轮做得快”，更体现在“每轮重启后重新找回状态的成本有多低”。

### 12. 这篇文章真正传达的是：长时 agent 需要的是一套工程交接机制，而不只是更长上下文

从整体看，这篇文章最有价值的地方不只是提出了 initializer agent 和 coding agent，而是揭示了 long-running agent 的本质设计原则：当任务跨越多个上下文窗口时，模型能力只是基础，真正决定系统是否能持续推进的，是有没有一套清晰的交接班机制、状态记录方式、完成定义和回退策略。也就是说，长时 agent harness 更像一个工程管理系统，而不是单纯的推理循环。

## 简要结论

如果把全文压缩成一句话，它想表达的是：`要让 agent 跨多个上下文窗口持续推进复杂工程任务，关键不是让它一次做更多，而是让它每次都只做一小步，并且把现场整理到足以让下一轮无缝接手。` 这篇文章最值得记住的不是某个具体文件名，而是它背后的思路：把长时 agent 设计成一种可交接、可回滚、可验证的连续工程流程。
