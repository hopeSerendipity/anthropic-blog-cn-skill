# Writing effective tools for agents - 笔记

原文标题：`Writing effective tools for agents — with agents`

## 文章主旨

这篇文章主要讲的是：`agent 的效果很大程度取决于工具设计，而不只是模型能力。` Anthropic 认为，很多 agent 表现不佳的问题，本质上不是“模型不够聪明”，而是工具不适合 agent 的认知方式和工作路径。文章给出了一条很清晰的方法论：先快速做出工具原型，再用真实任务构建评测，然后根据 transcript、指标和错误模式持续优化工具定义、工具返回结果和工具描述，必要时还可以让 Claude 反过来参与工具优化。

## 核心观点

### 1. 工具是“确定性系统”和“非确定性 agent”之间的契约

文章开头最重要的定义是，工具不再只是一个普通 API 包装层，而是连接确定性软件系统和非确定性 agent 的中间契约。传统函数在相同输入下会稳定地产生相同输出，但 agent 不会，它可能选择不同路径、误解工具、先问澄清问题，甚至直接产生幻觉。因此，工具设计的重点不只是“功能是否存在”，而是“agent 是否容易理解、愿意调用、并能正确调用”。这意味着为 agent 写工具时，设计思路必须和传统 API 设计明显不同。

### 2. 不要纸上谈兵，先做能跑的原型

文章强调，很多工具是否顺手，只有真正让 agent 用起来之后才能知道。因为一旦进入实际调用流程，很多原本抽象设计里看不见的问题就会暴露出来，比如参数名太模糊、工具边界不清、返回内容太长、或者调用方式不自然。这个观点非常实用，因为它说明工具的人机工程属性很难靠纯设计推断，最快的方式往往是先做出一个可运行的版本，再从真实使用中发现问题。

### 3. 评测必须贴近真实任务，不能只做漂亮的沙盒演示

Anthropic 明确反对过于简单的 sandbox 评测。文章认为，如果评测任务太短、太浅、太接近教科书命令，那么你优化出来的工具也只会适配这种人为简化的环境，而无法代表真实业务中的复杂任务。高质量评测应建立在真实数据源、真实服务和多步任务链上，并且要有可验证结果。这个观点很重要，因为它把“工具优化”从拍脑袋的主观改良，变成了基于真实工作流的工程闭环。

### 4. 不能只看最终对错，还要看中间过程

文章特别强调 transcript、reasoning、feedback 和工具调用指标的重要性，因为很多问题不会体现在最终答案是否正确上，而是体现在中间过程是否低效、是否反复犯错、是否总是走弯路。比如 agent 可能最后答对了，但中间浪费了大量 token、做了很多冗余调用，或者始终靠运气绕过去。这个观点提醒我们，评测 agent 工具使用质量时，不能只盯着准确率，还要分析它究竟是怎么完成任务的。

### 5. 工具不是越多越好，而是越贴近高价值工作流越好

文章反复强调，不要简单把现有 API 或软件能力逐个打包成工具，因为这很容易制造一堆低层、碎片化、彼此重叠的能力，反而让 agent 更难选、更难用。更好的做法是围绕高价值工作流设计少量高质量工具，比如直接做 `schedule_event`，而不是拆成 `list_users`、`list_events`、`create_event` 让 agent 自己拼。这个观点背后的本质是，agent 更需要“能推进任务的工具”，而不是“形式上能力完整的工具集合”。

### 6. 工具应该帮助 agent 减少上下文浪费

文章用通讯录的例子解释得很清楚：如果一个工具一次性返回所有联系人，agent 就不得不逐个阅读这些内容，浪费宝贵上下文；而更合理的方式应该是先缩小范围，再返回相关结果。这个观点的意义在于，它揭示了 agent 工具设计的核心约束之一并不是计算资源，而是上下文资源。对 agent 来说，很多工具设计问题本质上都是“如何避免把大量无关信息塞进上下文”的问题。

### 7. 工具边界和命名空间会直接影响 agent 选工具的能力

当 agent 接入几十个 MCP server 和几百个工具时，工具名本身就成了判断边界的重要线索。文章建议通过 namespacing，让工具按服务或资源分组，例如 `asana_search` 或 `asana_projects_search`，从而帮助 agent 更快地理解哪些工具属于同一类能力。更值得注意的是，Anthropic 还观察到前缀式还是后缀式命名会对不同模型带来不同影响，这说明命名不只是表面规范问题，而是会真实影响工具调用效果的设计变量。

### 8. 工具返回给 agent 的，应该是“高信号信息”

文章建议工具响应要优先返回和后续决策直接相关的信息，而不是默认把所有底层字段都吐出来。像 `name`、`image_url`、`file_type` 这类字段，通常比 `uuid`、`mime_type` 这类技术字段更能帮助 agent 做出下一步判断。这个观点非常关键，因为它说明工具不是单纯的数据转发层，而是在主动决定“哪些信息对模型最有用”，也就是在做一种面向 agent 的信息压缩与筛选。

### 9. 自然语言式标识通常比技术 ID 更适合 agent

文章指出，agent 更擅长处理带有语义的名称、术语和可解释标识，而不擅长处理随机 UUID 这类晦涩 ID。实践上，即使只是把内部标识转换成更容易理解的名字，或改成更规整的编号体系，也能显著提升检索精度并减少幻觉。这个观点很有启发性，因为很多系统默认把机器友好的标识直接暴露给 agent，但 agent 实际上更接近“语言推理系统”，而不是数据库客户端。

### 10. 响应粒度应该可控，而不是永远返回一种格式

文章提出了一个很实用的模式：给工具加上类似 `response_format` 的参数，让 agent 在 `concise` 和 `detailed` 之间自行切换。这样，agent 在只想理解内容时可以拿更短的结果，在确实需要继续触发下游调用时再拿包含 ID 和技术字段的详细结果。这个观点的价值在于，它同时兼顾了 token 效率和任务能力，也说明“少返回”并不是绝对目标，关键是要让响应粒度能根据任务阶段灵活变化。

### 11. token 效率不是附加优化，而是工具设计本身的一部分

文章把分页、范围限制、过滤、截断和错误提示都放到了设计原则层面，而不是实现细节层面。因为对 agent 来说，上下文就是稀缺资源，返回过量信息会直接拉低表现。更进一步，作者还强调，如果做了截断或输入校验失败，错误信息本身也要写得足够清楚，最好能直接告诉 agent 下一步应该怎么调整。这个观点非常重要，因为它说明一个好工具不仅要“能给结果”，还要“能把 agent 引导到更省 token、更稳定的行为路径上”。

### 12. 工具描述本身就是一种 prompt engineering

文章认为，提升工具效果最直接的方法之一，就是认真打磨工具描述和规格，因为这些文本会直接进入 agent 的上下文，进而影响它对工具用途、输入参数和适用场景的理解。写工具描述时，最好把 agent 当作一个新同事，把原本默认存在的术语定义、查询格式和资源关系全部显式写出来，并通过清晰参数命名来降低歧义。这个观点之所以重要，是因为它说明很多性能提升并不需要改代码，只要改对了描述，就可能显著改善工具调用质量。

### 13. Claude 不只是工具使用者，也可以是工具优化者

文章最后一个很强的观点是，可以把评测 transcript 直接交给 Claude Code，让它帮你分析工具定义矛盾、实现低效和描述不一致的问题。Anthropic 明确表示，文中很多结论其实就是通过这种“让 Claude 参与反向优化工具”的过程得来的。这意味着 agent 不只是需要被工具服务，它本身也可以进入工具改进闭环，成为调试和重构工具系统的一部分。

## 简要结论

如果把整篇文章压成一句话，它真正想表达的是：`高效 agent 依赖的不是更多工具，而是更适合 agent 工作方式的工具。` 这些工具应该边界清晰、返回高信号信息、节制使用上下文、允许灵活组合、并通过评测持续迭代优化。只要你把工具设计也纳入系统化评测闭环，agent 的能力上限就会明显提高。
