# Building agents with the Claude Agent SDK - 中文翻译

去年，我们曾结合与客户合作的经验，分享了构建高效智能体的一些心得。此后，我们发布了 Claude Code，这是一款最初为了提升 Anthropic 内部开发者生产力而打造的智能编程解决方案。

在过去几个月里，Claude Code 已经远远不止是一款编程工具。在 Anthropic，我们一直在用它做深度研究、视频制作、笔记整理，以及数不清的其他非编程工作。事实上，它已经开始驱动我们几乎所有重要的智能体循环。

换句话说，驱动 Claude Code 的智能体底座，也就是 Claude Code SDK，同样可以驱动许多其他类型的智能体。为了体现这一更广泛的愿景，我们决定将 Claude Code SDK 更名为 Claude Agent SDK。

在这篇文章中，我们会介绍为什么要构建 Claude Agent SDK，如何用它搭建你自己的智能体，以及我们团队在实际部署过程中总结出的最佳实践。

## 给 Claude 一台电脑

Claude Code 背后的关键设计原则是：Claude 需要拥有程序员每天都会使用的同样工具。它需要能够在代码库中查找合适的文件、编写和编辑文件、运行 lint、执行代码、调试、修改，然后在必要时反复进行这些操作，直到代码成功运行。

我们发现，只要让 Claude 能够通过终端访问用户的电脑，它就具备了像程序员一样写代码所需的一切。

但这也让 Claude Code 里的 Claude 在非编程任务上同样表现出色。通过赋予它运行 Bash 命令、编辑文件、创建文件和搜索文件等工具，Claude 可以读取 CSV 文件、搜索网络、构建可视化、解读指标，以及完成各种其他数字化工作。简而言之，就是打造能够使用电脑的通用型智能体。Claude Agent SDK 的核心设计原则，正是“给你的智能体一台电脑”，让它们像人类一样工作。

## 创建新类型的智能体

我们相信，给 Claude 一台电脑，会释放出构建更高效智能体的能力。例如，借助我们的 SDK，开发者可以构建：

金融智能体：理解你的投资组合和目标，并通过访问外部 API、存储数据、运行代码进行计算，帮助你评估投资。

个人助理智能体：通过连接内部数据源、跨应用追踪上下文，帮助你预订行程、管理日历、安排预约、整理简报等。

客服智能体：处理高模糊度的用户请求，例如客服工单；它们可以收集和审查用户数据、连接外部 API、回复用户，并在需要时升级给人工处理。

深度研究智能体：在大规模文档集合中开展综合研究；它们能够搜索文件系统、分析并综合多个来源的信息、跨文件交叉验证数据，并生成详细报告。

以及更多其他场景。归根结底，这个 SDK 为你提供的是一组基础能力，让你可以围绕任何想自动化的工作流构建智能体。

## 构建你的智能体循环

在 Claude Code 中，Claude 往往运行在一个特定的反馈循环中：收集上下文 -> 采取行动 -> 验证结果 -> 重复。

智能体通常都会遵循这样一个反馈循环：收集上下文 -> 采取行动 -> 验证结果 -> 重复。

这也是思考其他智能体及其所需能力的一种有用方式。为了说明这一点，我们将以“如何用 Claude Agent SDK 构建一个邮件智能体”为例展开说明。

## 收集上下文

在开发智能体时，你给它的不应只是一个提示词；它还需要能够自行获取和更新自身的上下文。SDK 中的以下特性可以帮助实现这一点。

### 智能体式搜索与文件系统

文件系统本身就代表了一类可以被拉入模型上下文的信息。

当 Claude 遇到大文件，比如日志文件或用户上传的文件时，它会决定如何借助 `grep`、`tail` 等 Bash 脚本把这些内容加载进自己的上下文。从某种意义上说，智能体的文件夹和文件结构本身，就是一种上下文工程。

例如，我们的邮件智能体可以把过去的对话存放在一个名为 `Conversations` 的文件夹中。这样，当被问及相关内容时，它就可以在这些历史对话中搜索所需上下文。

### 语义搜索

语义搜索通常比智能体式搜索更快，但准确性更差、维护更困难，也更不透明。它通常需要先把相关上下文切分成片段，为这些片段生成向量嵌入，再通过查询这些向量来搜索概念。考虑到这些局限，我们建议先从智能体式搜索开始，只有在你确实需要更快的结果或更多搜索变体时，再加入语义搜索。

### 子智能体

Claude Agent SDK 默认支持子智能体。子智能体主要有两个用途。第一，它们支持并行化：你可以同时启动多个子智能体处理不同任务。第二，它们有助于管理上下文：每个子智能体都使用自己独立的上下文窗口，只把相关信息返回给编排器，而不是把完整上下文全部发回。这使它们特别适合处理那种需要筛选大量信息、而其中大部分最终并无用处的任务。

在设计邮件智能体时，我们可以给它加入“搜索子智能体”能力。邮件智能体可以并行启动多个搜索子智能体，对你的邮件历史执行不同查询，然后只返回相关摘录，而不是整封邮件线程。

### 压缩

当智能体长时间运行时，上下文维护就变得至关重要。Claude Agent SDK 的 `compact` 功能会在上下文接近上限时自动总结之前的消息，从而避免智能体耗尽上下文空间。这个功能构建在 Claude Code 的 `compact` 斜杠命令之上。

## 采取行动

在收集到上下文之后，你还需要赋予智能体灵活采取行动的方式。

### 工具

工具是智能体执行能力的核心构件。工具会在 Claude 的上下文窗口中占据显著位置，因此也是 Claude 在决定如何完成任务时最优先考虑的行动方式。这意味着，你应该有意识地设计工具，以尽可能提升上下文效率。关于这方面的最佳实践，你可以参考我们的博客文章《用智能体为智能体编写高效工具》。

因此，你的工具应当是你希望智能体优先采取的关键动作。你可以学习如何在 Claude Agent SDK 中创建自定义工具。

对于邮件智能体，我们可以定义诸如 `fetchInbox` 或 `searchEmails` 这样的工具，作为它最核心、最常用的操作。

### Bash 与脚本

Bash 作为一种通用工具非常有用，它让智能体能够借助电脑以灵活方式完成工作。

在邮件智能体的场景中，用户的重要信息可能存放在附件里。Claude 可以通过编写代码来下载 PDF、将其转换为文本，再搜索其中内容以找到有价值的信息。

### 代码生成

Claude Agent SDK 非常擅长代码生成，这并非偶然。代码具有精确、可组合、可无限复用的特点，因此对于那些需要可靠执行复杂操作的智能体来说，它是一种理想的输出形式。

在构建智能体时，不妨思考一下：哪些任务如果用代码来表达，会更有优势？很多时候，这个问题的答案会解锁巨大的能力提升。

例如，我们最近在 Claude.AI 中推出的文件创建功能，完全依赖代码生成。Claude 会编写 Python 脚本来创建 Excel 表格、PowerPoint 演示文稿和 Word 文档，从而保证格式一致并实现复杂功能，而这很难通过其他方式做到。

在邮件智能体中，我们可能希望允许用户为收到的邮件创建规则。为了实现这一点，我们可以编写代码，并在对应事件发生时执行。

### MCP

模型上下文协议（MCP）为外部服务提供了标准化集成，能够自动处理身份认证和 API 调用。这意味着你可以把智能体连接到 Slack、GitHub、Google Drive 或 Asana 等工具，而无需自己编写集成代码，也不必手动管理 OAuth 流程。

对邮件智能体来说，我们可能希望搜索 Slack 消息来了解团队背景，或者查看 Asana 任务，确认是否已经有人被分配去处理某个客户请求。有了 MCP 服务器，这些集成几乎可以开箱即用。你的智能体只需调用诸如 `search_slack_messages` 或 `get_asana_tasks` 之类的工具，而具体的集成细节由 MCP 负责。

随着 MCP 生态不断壮大，你可以快速为智能体增加新的能力，把重点放在智能体行为本身，而不是集成工作上。

## 验证你的工作

Claude Code SDK 通过评估自身产出，完成智能体循环的最后一步。能够检查并改进自身输出的智能体，本质上更可靠。它们会在错误进一步扩大之前就发现问题，在偏离目标时进行自我纠正，并在反复迭代中持续改进。

关键在于，要给 Claude 提供具体的方法来评估自己的工作。以下是我们发现比较有效的三种方式。

### 定义规则

最好的反馈形式，是为输出提供清晰定义的规则，并说明哪些规则失败了、为什么失败。

代码 lint 就是一种非常优秀的基于规则的反馈形式。反馈越深入越好。例如，通常来说，生成 TypeScript 并运行 lint，往往比生成纯 JavaScript 更好，因为前者会提供更多额外的反馈层次。

在生成邮件时，你可能希望 Claude 检查邮箱地址是否有效；如果无效，就抛出错误。你也可能希望它检查用户之前是否给这个地址发过邮件；如果发过，就抛出一个警告。

### 视觉反馈

当你使用智能体完成视觉任务时，比如生成或测试 UI，视觉反馈（如截图或渲染结果）会非常有帮助。比如在发送 HTML 格式邮件时，你可以先把生成的邮件截图，再把截图返回给模型做视觉验证和迭代优化。模型随后会检查视觉输出是否符合请求。

例如，它可以检查：

布局：元素位置是否正确？间距是否合适？

样式：颜色、字体和格式是否符合预期？

内容层级：信息是否按照正确顺序呈现，重点是否突出？

响应式表现：页面是否显得破碎或拥挤？不过单张截图对视口信息的覆盖有限。

借助像 Playwright 这样的 MCP 服务器，你还可以自动化这个视觉反馈循环：为渲染后的 HTML 拍摄截图、捕捉不同视口尺寸下的效果，甚至测试交互元素，并把这一切纳入智能体工作流中。

Claude 会对由智能体生成的邮件正文提供视觉反馈。来自大语言模型的视觉反馈，可以为你的智能体提供有帮助的指导。

### 让 LLM 担任评审

你还可以让另一个语言模型根据一些模糊规则来“评判”智能体的输出。不过，这种方法通常不算特别稳健，而且在延迟成本上也可能比较高。但在某些应用场景中，只要能带来一点性能提升，就值得付出额外成本。

例如，我们的邮件智能体可以让一个独立的评审子智能体检查草稿的语气，看它是否与用户此前的邮件风格一致。

## 测试并改进你的智能体

当你把智能体循环跑过几轮之后，我们建议你开始测试智能体，确认它是否真的具备完成任务所需的能力。改进智能体的最好方法，就是仔细观察它的输出，尤其是它失败的案例，并设身处地站在它的角度去思考：它手上是否真的有完成这项工作的合适工具。

在评估你的智能体是否准备充分时，还可以问自己以下问题：

如果你的智能体误解了任务，可能是因为它缺少关键信息。你能否调整搜索 API 的结构，让它更容易找到自己需要知道的内容？

如果你的智能体反复在某项任务上失败，能否在工具调用中加入一条正式规则，用来识别并修复这种失败？

如果你的智能体无法自行修复错误，能否给它提供更有用、或更有创造性的工具，让它换一种方式解决问题？

如果你的智能体随着功能增加而出现性能波动，那就应该基于客户的真实使用情况，构建一个具有代表性的测试集，进行程序化评估，也就是 evals。

## 开始上手

Claude Agent SDK 通过让 Claude 访问一台电脑，从而更容易构建自治智能体。在这台电脑上，它可以写文件、运行命令，并反复迭代自己的工作。

围绕这个智能体循环来思考，也就是收集上下文、采取行动以及验证结果，你就可以构建出既可靠、又易于部署和迭代的智能体。

你现在就可以开始使用 Claude Agent SDK。对于已经基于该 SDK 构建应用的开发者，我们建议按照迁移指南升级到最新版本。
