# Effective context engineering for AI agents - 笔记

发布时间：`2025-09-29`

## 文章主旨

这篇文章主要讲的是，随着 agent 从一次性 prompt 驱动的任务，转向多轮推理、长期运行、持续调用工具的工作流，真正决定系统表现的重点已经不再只是“prompt 写得好不好”，而是“在每一步里，到底给模型什么上下文”。Anthropic 把这种问题定义为 `context engineering`：在有限且会逐渐失效的上下文窗口里，持续筛选出最小但信号最强的信息组合，从而稳定地产生目标行为。文章同时给出了一套比较完整的框架，包括 system prompt 的抽象高度、工具与示例如何保持精炼、运行时如何做 just-in-time 检索，以及长时任务里如何通过 compaction、记忆和多 agent 结构绕开上下文限制。

## 核心观点

### 1. `Context engineering` 是 `prompt engineering` 的自然升级

文章一开始最重要的判断是，今天构建 agent 的问题，已经不能只理解为“把 prompt 写得更好”，而必须理解成“如何持续配置模型看到的整体上下文”。`Prompt engineering` 关注的是指令怎么写，而 `context engineering` 关注的是 system prompt、工具、外部数据、消息历史、MCP、运行时检索结果等所有会进入上下文的内容该如何共同组织。这个观点很关键，因为它把工程重点从“优化单条指令”转移到了“管理一个动态变化的信息状态”。

### 2. 上下文必须被当成一种有限资源，而不是越多越好

文章强调，LLM 虽然能支持越来越长的上下文，但它们并不会因为能“装下”更多信息，就自动更擅长使用这些信息。随着 token 增长，模型检索和回忆关键信息的能力会逐渐下降，这就是文中提到的 `context rot`。因此，上下文的核心矛盾不是容量不够，而是注意力预算有限。这个观点很重要，因为它提醒我们，往上下文里继续堆内容，收益往往递减，甚至可能直接让系统变差。

### 3. 好的上下文不是“更多信息”，而是“尽可能小但足够高信号的信息集合”

Anthropic 给出的核心原则非常清楚：应努力找到最小的一组高信号 tokens，使其最大化产生目标行为的概率。这里的重点不是机械地追求短，而是追求高信息密度和高相关性。这个观点的价值在于，它为几乎所有 agent 设计决策提供了统一标准：system prompt 要不要再加一段、工具结果要不要全返回、示例要不要继续堆，最终都可以回到“它是不是在消耗有限注意力预算，却没有显著增加有效信号”这个判断上。

### 4. system prompt 的关键不只是清楚，还要处在合适的“抽象高度”

文章对 system prompt 的建议非常实用：一端是把行为逻辑写成脆弱的 if-else 风格硬编码，另一端是给出空泛且假设共享背景的高层描述，二者都不好。理想状态是处在一个“既足够具体，又不至于僵硬”的中间层。这一观点很有价值，因为很多 prompt 失败并不是因为信息少，而是因为信息所处的抽象层级不对，要么过细导致脆弱，要么过粗导致模型根本抓不到可执行信号。

### 5. 工具设计本质上也是上下文设计的一部分

文章把工具放在 context engineering 里来讨论，而不只是当成功能扩展，这一点非常值得记住。因为工具不仅决定 agent 能做什么，也决定它会看到什么信息、会怎么在任务过程中拉入更多上下文。如果工具集合臃肿、边界重叠、参数模糊，agent 就不仅更难做决策，还会把大量无关信息带进上下文。这个观点的本质是，工具不是独立于上下文之外的外部组件，而是上下文流动机制本身的一部分。

### 6. 示例应当是“代表性行为样本”，而不是边缘规则大杂烩

文章延续了 Anthropic 一贯的观点：few-shot 例子很有用，但不要试图把所有边缘情况都塞进去。更好的做法是挑一组多样化、标准化、具有代表性的 canonical examples，让模型从样本中学到预期行为。这个观点很重要，因为它说明示例不是规则文档的替代品，而是一种高密度行为信号。示例的价值不在覆盖一切，而在于高效传达模式。

### 7. agentic search 的核心思想，是把“所有可能相关信息”变成“可在运行时按需加载的信息引用”

在上下文检索部分，文章提出了一个很重要的转向：从预先把大量相关数据取出来，转向保留轻量级引用，例如文件路径、查询、链接，再让 agent 在运行中按需调工具加载信息。这个观点很关键，因为它说明更强的 agent 不一定依赖更重的预取，而是更依赖“只在需要时看需要的东西”。这是一种把上下文从静态准备转向动态装配的思路。

### 8. 文件路径、目录层级、命名和时间戳，本身也是上下文信号

文章特别提到，agent 在文件系统里导航时，文件名、层级结构和时间戳等元数据本身就能提供强烈语义线索。例如 `tests/test_utils.py` 和 `src/core_logic/test_utils.py` 的含义完全不同。这个观点很有启发性，因为它说明有价值的上下文不一定非得来自文件内容本身，结构化元数据同样能大幅帮助 agent 判断下一步该看什么、怎么理解某段信息。

### 9. 渐进式披露比一次性灌入更适合复杂探索型任务

文章把 agent 自主探索环境描述为一种 `progressive disclosure` 机制，也就是让它每次只接触一小层信息，再根据这层信息决定下一步。文件大小、命名风格、时间信息等都会逐层塑造 agent 的理解。这个观点的重要性在于，它揭示了复杂任务中的一个普遍规律：把一切一次性塞给模型，往往不如让模型通过逐步探索来自我构建理解，因为后者更符合有限工作记忆下的有效推理方式。

### 10. 对长时任务来说，窗口变大并不能自动解决上下文问题

在长时任务部分，文章明确反对一种直觉：只要未来上下文窗口更大，问题自然就会消失。Anthropic 的判断是，即使窗口继续扩大，上下文污染和信息相关性仍然会长期存在，所以长时任务的核心挑战不会自动消失。这个观点很重要，因为它说明 long-horizon agent 的问题本质上不是单纯的“容量不足”，而是“如何让长期运行过程中始终保留清晰而高价值的状态表示”。

### 11. `Compaction`、`note-taking` 和 `sub-agent` 是三种不同的长期状态管理手段

文章把长时任务的主要策略拆成三类：`compaction` 适合在上下文快满时做高保真压缩并续上会话；`structured note-taking` 适合把关键信息持久化到上下文外，再在需要时读回；`sub-agent architectures` 则适合把大任务拆给多个干净上下文的专门 agent，再由主 agent 汇总。这个观点很有价值，因为它说明长时任务没有单一通用解法，而是要根据任务特征选择不同的上下文管理机制。

### 12. 这篇文章真正传达的是一种更高层的方法论：把 agent 看成“在有限注意力预算下持续配置信息流的系统”

从整体上看，这篇文章最有价值的地方并不是某个局部技巧，而是它给出了一种更成熟的 agent 设计观。一个强 agent，不只是一个 prompt 写得好的模型，也不是一个工具很多的系统，而是一个能在每一步都认真决定“什么该进上下文、什么该留在外部、什么该压缩、什么该延后加载”的信息流系统。这个观点几乎可以作为全文的总纲，因为它把 prompt、工具、检索、记忆和多 agent 架构都统一到了同一个框架下。

## 简要结论

如果把全文压缩成一句话，它想表达的是：`构建强大 agent 的关键，不只是告诉模型该做什么，而是在每一步都只给它最值得注意的那部分信息。` 这篇文章的最大价值，是把 prompt、工具、检索、记忆和长时任务管理统一成了一个问题来看：如何在有限上下文里持续维护高信号、低噪声的状态。
