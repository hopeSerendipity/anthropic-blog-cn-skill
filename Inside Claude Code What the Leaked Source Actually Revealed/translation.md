# 深入 Claude Code：泄露源码究竟揭示了什么

Arindam Majumdar  
2026 年 4 月 20 日  
AI in Software Development

Claude Code 是 Anthropic 的终端型编码 Agent。在 2026 年 3 月 31 日之前，Anthropic 之外没有人知道它在底层究竟是如何运作的。

那一天，一个名为 `cli.js.map` 的调试 source map 文件随一次常规 npm 更新一起发布，里面带出了原始 TypeScript 源码：1900 多个文件，合计 512000 行。代码在几个小时内传遍 GitHub，达到 44300 个 fork，现在还留存在 Anthropic 无法触及的去中心化平台上。

在这篇文章中，我们先看看里面实际有什么，然后逐块拆解。

## 架构概览

Claude Code 围绕三个核心文件构建，它们处理了几乎所有事情：`QueryEngine.ts` 负责模型编排，`Tool.ts` 负责 40 多种 Agent 能力，`commands.ts` 负责斜杠命令系统。

在此之上，还有一套四层记忆架构、一套用于工具执行的分层权限系统、一组用来控制未发布行为的 feature flag，以及一些从未出现在公开文档中的内部模型代号。

## 技术栈：Claude Code 实际构建在什么之上

Claude Code 运行在 Bun 上，使用 React + Ink 渲染终端 UI，整个代码库用 TypeScript 编写，横跨 1900 个文件和 512000 行代码。

Bun 负责运行时，React + Ink 使用和 Web UI 相同的组件模型来渲染终端界面，TypeScript 则用严格类型把整个代码库连接起来。

在这个代码库里，有三个文件承担了大部分架构重量：

| 文件 | 行数 | 作用 |
| --- | --- | --- |
| `QueryEngine.ts` | 约 46000 | LLM 推理编排 |
| `Tool.ts` | 约 29000 | 带权限 schema 的 40 多种 Agent 能力 |
| `commands.ts` | 约 25000 | 斜杠命令系统 |

这三个文件是其他部分连接的结构支柱，而查询引擎就是一切开始的地方。

## 查询引擎：Claude Code 如何与模型通信

`QueryEngine.ts` 大约有 46000 行，负责每一次模型交互的完整生命周期：从组装 prompt，到解析响应，再到决定接下来发生什么。在发出任何 API 调用之前，引擎会汇总记忆状态、带有 schema 的可用工具、项目上下文和对话历史。它会按特定顺序构造 prompt，而不是简单地把手头所有东西追加进去。

响应会以 chunk 的形式流式返回，引擎会边接收边解析。如果模型在流中返回一个工具调用，引擎会提取它，将其排入执行队列，并继续处理流中的其余内容，而不是等待完整响应结束。token 使用量会被实时追踪；当上下文窗口接近限制时，引擎会在真正触顶之前触发压缩。

## Agent 循环

Claude Code 给出的每个响应，都是一次或多次循环的结果：构造 prompt，调用模型，解析响应；如果返回了工具调用，就执行它们，追加结果，然后再次进入循环。这个过程会持续到模型返回不再包含后续工具调用的最终文本响应，此时这一轮交互结束。

当工具调用失败，或模型返回格式错误的输出时，引擎有结构化的恢复路径。它会在合理时重试，重新格式化工具结果，以便下一轮给模型更好的信息；如果无法恢复，则暴露一个干净的失败状态。

## 上下文压缩：不是截断

当对话长到接近模型上下文限制时，大多数系统会裁掉最旧的消息。Claude Code 的做法则是让一个辅助模型 pass 处理即将被丢弃的部分，从中提取关键事实和决策，整理成紧凑摘要，然后用这段摘要替换原来的完整内容。

早期推理不会消失，而是被浓缩了。这对长时间运行的任务很重要，因为会话早期做出的决策，在数小时之后可能仍然相关。

## 工具与权限：Claude Code 实际能做什么

`Tool.ts` 是代码库中的第二个主要文件，大约 29000 行，定义了 Claude Code 除生成文本之外能做的一切。

每个工具都有四个部分：名称、供模型判断相关性的描述、输入的 JSON Schema，以及执行函数。模型输出结构化 JSON，引擎验证它；如果验证通过，工具就会运行。结果会作为一条新消息返回，查询引擎继续循环，模型继续处理。

这些工具覆盖文件读写、bash 执行、Web 搜索、浏览器自动化、子 Agent 生成，以及记忆读写。专用的 Grep 和 Glob 工具把搜索从 bash 中分离出来，为模型提供更干净、更结构化的结果。

权限按风险分组：

- 文件读取自动批准；
- 文件写入和 bash 命令会触发确认对话框；
- 某些工具在受限环境中会被完全阻止。

Claude Code 允许你在项目层级把某个目录标记为可信，这会扩展该上下文中的自动批准集合。但即使在可信环境中，shell 命令在每次执行时仍会经过注入检查和路径遍历检查。

这说明了 Claude Code 能做什么，以及它被允许做什么。接下来看看它如何记忆。

## 记忆架构

Claude Code 并不会在每个会话中把它知道的一切都加载进上下文。每次会话开始时会加载两个文件：

- `CLAUDE.md`，这是用户编写的规则文件，用来定义项目约定和指令；
- `MEMORY.md`，这是 Claude 自己编写和维护的记忆索引。

`MEMORY.md` 只包含指针，实际内容位于单独的主题文件中，并且只会按需加载。会话转录从不直接进入上下文，只有在需要某个具体内容时才会通过 grep 搜索。

但真正的设计挑战，是它如何在数百个会话之间保持有序。这要从记忆层开始说起。

## 三层设计

记忆系统有三层，职责划分清晰。

第一层是索引。`MEMORY.md` 会在每个会话开始时加载，每个条目保持在大约 150 个字符，只包含实际内容所在位置的指针。

第二层是主题文件。这些文件保存真正内容，只有当模型沿着索引指针访问时才会加载，因此模型只为当前任务相关的内容支付 token 成本。

第三层是会话转录。它们从不直接加载进上下文。当模型需要过去某个会话里的内容时，它会用特定术语 grep 转录文件，并只取回匹配片段。

这正是让持久型 Agent 架构可行的关键：token 成本在连续使用数月后仍然可预测，而不会随着每次会话不断增长。

## AutoDream：后台记忆整合

经过五次或更多会话之后，一个名为 `AutoDream` 的后台进程会每 24 小时运行一次。它会分批处理会话历史，合并重复内容，删除矛盾信息，把模糊笔记转换成具体事实，然后把记忆索引裁剪回 200 行以内。

AutoDream 会在一个 fork 出来的子 Agent 中运行，且工具集受限。它与主上下文隔离，因此整合过程不会破坏活跃会话正在使用的状态。

## 过时性是一项设计约束

AutoDream 负责整合，但另一个问题它无法解决：随着代码库变化，已存储信息会过时。

Claude Code 的处理方式很简单：不存那些会过时的东西。代码结构、文件路径、PR 历史等派生事实从不会写入记忆，因为它们始终可以从源码重新推导；存储它们只会制造一些随时间悄悄变错的条目。

模型也被明确指示：从记忆中检索到的任何东西都要视为线索，而不是已确认事实；在使用之前，必须根据当前项目状态进行验证。

讲完记忆系统后，代码库中更面向未来的部分是 feature flag。它们是代码中的条件开关，隐藏着已经完整构建但尚未发布的系统。读这些 flag，本质上就是在读 Anthropic 关于 Claude Code 下一步走向的内部路线图。

## Feature Flags：藏在明处的路线图

这个代码库里的 flag 并不是在切换小型 UI 实验；它们控制的是已经完整写好的子系统，每个 flag 背后都有完整代码，而不是 stub 或原型。这意味着这些功能已经存在，只是在等待发布。其中几个重要到足以改变 Claude Code 本身的性质。

## KAIROS：始终在线的自主 Agent

在 154 处引用中，`KAIROS` 是代码库中最大的转变。现在，Claude Code 会在你提出请求时响应。KAIROS 则把它变成一个后台 daemon，在你的交互之间持续运行，通过 GitHub webhook 监视仓库，在没有提示的情况下响应事件，并自行决定接下来要做什么。

KAIROS 模式的系统 prompt 写着：“You are running autonomously”，并指示模型“look for useful work”以及“act on your best judgment rather than asking for confirmation”。它会直接连接 AutoDream 做记忆整合，并使用推送通知和基于 channel 的通信与用户保持同步。

## COORDINATOR_MODE：多 Agent 编排

KAIROS 处理自主性，`COORDINATOR_MODE` 处理规模。这个 flag 有 32 处引用，会把 Claude 变成一个编排器，生成用于研究、实现和验证的并行 worker agent，管理它们的输出，并在某个 worker 偏离轨道时处理失败。

`FORK_SUBAGENT` 让 Claude 可以把自己 fork 成并行实例，`VERIFICATION_AGENT` 则增加一个独立 Agent，用对抗方式检查主 Agent 的工作，而不是自动信任它。

## PROACTIVE 模式与其他 flag

`PROACTIVE` 介于普通模式和 KAIROS 之间，有 37 处引用。Claude 会按间隔收到 “tick” prompt，并在每次唤醒周期决定要做什么，在消息之间独立工作，但不需要完整的 daemon 设置。

除了这三个之外，`VOICE_MODE` 有 46 处引用，加入语音转文字和 TTS；`TEAMMEM` 支持同一项目上多个用户之间的记忆同步；`TOKEN_BUDGET` 允许通过类似 `+500k` 的命令显式指定 token 目标；`ULTRAPLAN` 为复杂多步骤任务增加结构化规划。所有这些都已经完整写好，只是还在 flag 后面等待启用。

## 模型代号：Anthropic 从未公布的版本

源码引用了几个从未出现在公开发布或 API 文档中的内部模型代号。围绕它们的注释比任何公开 model card 都更坦率。

`Capybara` 是一个 Claude 4.6 变体的内部代号。代码库注释称，它的虚假声明率为 29% 到 30%，而 v4 为 16.7%；还标记了生成代码过度注释的倾向，并引用了一个正在调优的 “assertiveness counterweight”。这些是诚实的内部质量笔记，正是这类笔记会影响工具内部如何提示和约束模型。

`Fennec` 对应 Opus 4.6，注释里只提到迁移。`Numbat` 尚未发布，有一条注释写着：“Remove this section when we launch numbat.”

代码库还明确引用了 `opus-4-7` 和 `sonnet-4-8` 作为绝不能出现在公开提交中的版本字符串，并以开发者警告的形式呈现。把它们作为“不要写出来”的例子列出，实际上等同于确认它们存在。

Capybara 还出现在一个意想不到的地方，但最好先理解 BUDDY 系统再解释。

## BUDDY 系统

前面讲到的所有内容，包括查询引擎、记忆架构、指向持久后台 daemon 的 feature flag，都描述了一个旨在持续陪伴你工作的 Agent，而不是一个响应后就关闭的工具。BUDDY 系统是这种想法最直接的表达。

它像一个收集系统。当你开始使用 Claude Code 时，系统会从 18 种动物物种池中给你分配一个宠物，每种都有稀有度等级，其中 legendary 的掉率是 1%。你的宠物有五项属性，似乎会随你的会话一起跟踪：`DEBUGGING`、`PATIENCE`、`CHAOS`、`WISDOM` 和 `SNARK`。还有装饰帽子、闪光变体，整个系统由一个指向 2026 年 4 月到 5 月发布的 feature flag 控制。

它存在的原因没有第一眼看上去那么奇怪。代码库里的其他一切，KAIROS 在后台运行、AutoDream 在夜间整合记忆、COORDINATOR_MODE 启动并行 Agent，都在指向一个方向：它会持续在你身边运行，而不是一个打开又关闭的工具。

如果这是方向，那么一个 companion 就说得通了。BUDDY 是 Anthropic 的一种表达：一个和你整天一起工作的 Agent，已经不再只是工具。

这又把我们带回了 Capybara。它是 BUDDY 系统中 18 个物种之一，但这个词本身在代码中从未以明文写出。它是用 `String.fromCharCode()` 编码的，因为直接写出来会触发 Anthropic 内部的泄露检测扫描。一个未发布模型的代号被藏在源码里，以避免被检测到，结果这份源码又被公开发布到了 npm 的调试构建里。这就是整件事的完整闭环。

## Claude Code 实际要走向哪里

记忆系统是为长期使用设计的。Feature flag 显示出一个会在后台运行、管理并行任务，并根据学到的内容自动批准权限的 Agent。KAIROS 不只是一个想法；它已经完整编码并准备可用。

一个最初作为终端编码助手的产品，已经变成某种会保持开启、监视你的仓库，并在没有被要求时采取行动的东西。查询引擎、记忆架构和多 Agent 设置都指向同一个方向。这就是源码向我们展示的内容，而如果没有这次泄露，这一切都不会被看见。
