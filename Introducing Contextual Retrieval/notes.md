# Introducing Contextual Retrieval - 笔记

发布时间：`2024-09-19`

## 文章主旨

这篇文章主要讲的是，传统 RAG 在把文档切成小块之后，往往会把每个 chunk 原本依赖的大上下文切掉，结果导致系统虽然“检索到了文本块”，却未必能检索到真正有用的那一块。Anthropic 提出的 `Contextual Retrieval`，本质上是在做 embedding 和建立 BM25 索引之前，先给每个 chunk 补上一小段只属于它的上下文说明，从而让检索阶段更容易理解这块文本到底处在哪个文档、哪个主题和哪个时间语境里。文章的核心结论是，这种做法能明显降低检索失败率，而再叠加 reranking 后效果更强。

## 核心观点

### 1. 传统 RAG 的主要问题之一，不是“找不到语义”，而是“丢掉了局部文本依赖的大上下文”

文章指出，传统 RAG 把文档切 chunk 之后，每块文本通常只保留局部内容，但很多事实本身离不开上下文才能成立。例如一句“公司收入比上一季度增长了 3%”，如果没有公司名和时间范围，这句话即使被检索到了，也很难知道它是不是回答了用户的问题。这个观点很重要，因为它说明传统 RAG 的失败不一定来自 embedding 不够强，而可能来自输入给 embedding 的 chunk 本身已经失真了。

### 2. `Contextual Retrieval` 的本质，是在检索前给每个 chunk 补回“它是谁、来自哪里、处于什么语境”

Anthropic 提出的方案并不复杂：先用 Claude 根据整篇文档，为每个 chunk 生成一小段针对性的上下文说明，再把这段说明拼接到 chunk 前面，然后再做 embedding 和 BM25 索引。这样一来，原本孤立的一块文本就重新带上了来源、主题、时间段、相关实体等关键信息。这个观点的价值在于，它把检索优化的重点从“换更强的检索器”转向了“先把被检索对象变得更容易被检索”。

### 3. 这不是简单的摘要，而是“面向检索的 chunk 定位信息”

文章特别提到，他们也试过一些其他思路，例如给 chunk 加通用摘要、做 hypothetical document embedding 或基于摘要建立索引，但效果有限。`Contextual Retrieval` 与这些方法的区别在于，它不是给 chunk 加一段泛泛的文档总结，而是专门补充“这个 chunk 在整篇文档里应该如何被理解”的定位信息。这个差异很关键，因为检索所需的上下文并不等于更长的摘要，而是更有针对性的辨识线索。

### 4. `Contextual Embeddings` 和 `Contextual BM25` 结合起来，说明这个思路同时适用于语义检索和词法检索

文章没有只在 embedding 一侧使用上下文，而是同样把补了上下文的 chunk 拿去建立 BM25 索引。这一点很有意思，因为它说明“补回上下文”不仅能帮助向量检索理解语义，也能让 BM25 这种词法匹配方法更容易命中正确 chunk。换句话说，这不是针对某一种检索器的小技巧，而是一种对检索对象本身的普适增强。

### 5. 小知识库不一定需要 RAG，直接塞进 prompt 可能更简单

文章开头专门强调，如果知识库本身小于 200,000 tokens，那么最简单的做法通常就是直接把整个知识库放进 prompt，不一定非要上 RAG。再加上 Claude 的 prompt caching，这种方案在速度和成本上也变得更可接受。这个观点很有价值，因为它提醒我们不要把 RAG 当成默认答案，先判断数据规模是否真的已经大到必须引入检索流程，往往更工程化。

### 6. `Contextual Retrieval` 之所以可落地，一个关键原因是可以用 Claude 自动为 chunk 生成上下文

如果每个 chunk 都要手工补注释，那么这个方法几乎不可操作。Anthropic 的关键做法是直接让 Claude 基于整篇文档自动生成一段 50 到 100 tokens 的 chunk-specific context，然后把它拼接到 chunk 前。这个观点的意义在于，它把原本看似昂贵的人工整理工作转化成了可批量自动化的预处理流程，也说明大模型本身不仅能消费检索结果，还能参与改造检索系统。

### 7. Prompt caching 让这种预处理成本下降到一个可接受水平

文章特别强调，Claude 的 `prompt caching` 让你不必为每个 chunk 都重复传整篇文档，而是先缓存整篇文档，再让模型逐块生成上下文说明。这样，`Contextual Retrieval` 的一次性预处理成本会低很多。这个观点很实用，因为它说明这套方案并不只是“理论上有效”，而是已经被设计成可以在真实大知识库场景里以较低成本跑起来。

### 8. 加上下文后，检索失败率的下降幅度是很实质的

实验结果显示，单独使用 `Contextual Embeddings` 就把 top-20 检索失败率从 5.7% 降到 3.7%，如果进一步结合 `Contextual BM25`，则降到 2.9%。这不是细微优化，而是相当显著的下降。这个观点的关键意义在于，它证明“先把 chunk 讲清楚再去检索”并不是直觉层面的改进，而是可以在多领域数据集上稳定测出来的性能提升。

### 9. reranking 和 contextual retrieval 是可叠加的，而不是替代关系

文章最后又加了一层 reranking，并观察到失败率进一步降到 1.9%，总降幅达到 67%。这说明 `Contextual Retrieval` 并不是要替代传统 RAG 里的 reranker，而是和它形成前后两层优化：前者先让候选 chunk 更容易被找对，后者再从大量候选中精细筛掉不那么相关的部分。这个观点很值得记，因为它体现了一种工程上很重要的思想：检索系统的提升通常来自多层小改进的叠加，而不是某一个单点魔法。

### 10. 这篇文章真正传达的，是“优化检索不只是在调检索器，也是在改造被检索内容”

从更高层看，这篇文章最有价值的地方在于它改变了对 RAG 优化的思考方式。很多人优化 RAG 时，只会想到换 embedding 模型、调 chunk size、换 vector DB、加 reranker，但 Anthropic 这里强调的是：如果 chunk 本身脱离语境，那无论后面的检索器多强，系统都可能从一开始就站在错误输入上。这个观点非常重要，因为它把“内容表示方式”重新提到与“检索算法”同等重要的位置。

## 简要结论

如果把全文压缩成一句话，它想表达的是：`RAG 检索失败的一个关键原因，是 chunk 在切分后失去了原本依赖的语境，而 Contextual Retrieval 通过在检索前补回这层语境，显著提升了检索准确率。` 这篇文章的最大价值，不只是提出了一个新技巧，而是提醒我们：要提升 RAG，既要优化“怎么找”，也要优化“被找的内容长什么样”。
